### merge
- RTKQ는
	- 서버가 source of truth라고 가정함
	- 모든 refetch는 클라의 캐시 데이터를 대체한다고 가정
- pagination처럼 시간 지나면서 기존 캐시데이터와 API 응답을 합치는게 유용한 사례있음
- merge는
	- immer 기반의 mutation을 가능하게 해준다.
	- ==key당== 각 response가 기존 캐시 엔트리를 대체할거라고 가정함
	- merge는 `serializeQueryArgs`, `forceRefetch` option과 같이 쓰여야함

### serializeQueryArgs
- RTKQ는
	- 항상 cache key value를 시리얼라이즈한다. 
	- 그리고 캐시 엔트리에 저장하여 실제 키로 사용한다
	- 기본 시리얼라이즈는 endpoint이름 + primitive value or stable-serialize object
	- ``state.api.queries['getPokemon("pikachu")']`.`
	- 원래 createApi레벨에서 시리얼라이제이션 동작 정의가능
	- 이제 각 enpoint는 `serializeQueryAtgs` 를 정의할 수 있음
- 이 옵션은 2가지 메인 목적이 있음
	- endpoint에서 전달되었지만 실제로 "키"의 일부가 아닌 값(소켓 또는 클라 인스턴스와 같은)을 제외
	- endpoint당 single entry를 사용하여 cache key 동작 변경하기 (인피니트 스크롤)
```ts
 getPost: build.query<Post, { id: string; client: MyApiClient }>({
      queryFn: async ({ id, client }) => {
        const post = await client.fetchPost(id)
        return { data: post }
      },
      serializeQueryArgs: ({ queryArgs, endpointDefinition, endpointName }) => {
        const { id } = queryArgs
        // This can return a string, an object, a number, or a boolean.
        // If it returns an object, number or boolean, that value
        // will be serialized automatically via `defaultSerializeQueryArgs`
        return { id } // omit `client` from the cache key

        // Alternately, you can use `defaultSerializeQueryArgs`:
        // return defaultSerializeQueryArgs({
        //   endpointName,
        //   queryArgs: { id },
        //   endpointDefinition
        // })
        // Or  create and return a string yourself:
        // return `getPost(${id})`
      },
    }),
```

### forceRefresh

RTQK가 시리얼라이즈 쿼리 arg가 변경하지 않고 이미 cache entry가 이행될지라도 너가 강제로 refetch 하고싶을 수 있다.

무한스크롤 시나리오가 유즈케이스다. endpoint에 대한 one cache entry있고 다른 페이지 번호가 쿼리 인자로 제공되면 들어오는 응답값이 병합되어야하는 시나리오.

# 심천보님의 테스트 4.13

## What
#### 명세(Application Design)를 테스트 하라.

협력과 계약
- 대부분의 코드는 다른코드와 협력
- 협력에 필요한 것은 __어떻게?__ 가 아닌 __무엇__ 

#### 인터페이스 
- 인터페이스는 무엇을 표현
- 클라 코드에게 반드시 필요한 정보
- 협력하는 코드 사이의 계약
- 테스트 코드 또한 하나의 클라이언트

#### 정보은닉
- 공개된 인터페이스로만 협력
- 협력에 대한 구체적인 구현은 은닉
- 테스트 코드가 호출시에는?
- 당연히 테스트 코드한테도 은닉되어야 함.

## How
- 테스트하기 쉬운 코드 === 좋은 코드
조건
1. 같은 입력에 항상 같은 결과를 반환하는 코드 (결정적)
2. 외부 상태를 변경하지 않는 코드 (부수효과 없는)

## 좋은 테스트를 위한 Guide

**Mock** 의 위험성
- 테스트가 SUT 구현에 의존
	- Mock을 사용한 구현 기술의 강제를 꾀할 수 있음.
- 상세구현과 명세를 기반으로 구현해서 테스트코드는 꺠지면 안됨.
- 하지만 Mockist가 짠 TC는 깨질 위험이 많다.
	- 왜냐면 Mock으로 통제를 엄청 하고 있기 때문에
- 그럼 Mock 쓰지말까?
	- 필요한 경우가 있다. 

## 코드는 최대한 겸손해야 한다.
겸손의 의미는 테스트 할 수 없는 코드는 테스트가 필요 없는 만큼 단순해야 한다.

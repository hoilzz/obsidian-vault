- 기존 페이지웹의 빌링 인증 방식?
	- POST 형태로 sToken, useruid or accessToken, appUserId  보내주는 것 같다
- 현재 NEXT에서는 앱에서 웹뷰 접근시 쿠키를 이용하는가?
	- 헤더의 X-SToken, X-UserId를 이용한다
	- 이 때 accessToken과 appUserId는 쿼리로 받을 수 있다.
		- 회원가입 시점에 stoken, userid가 없고 카카오에서 발급받은 저것들만 있으니 쿼리로 받는다.
		- 가입 시점에 stoken과 userId가 없으니 전달받은 accessToken, appUserId로 jwt를 만듭니다.
		- 웹뷰에 sToken과 userId가 없는데 at, appUserId를 전달받았다면 요걸로 로그인 API 호출하여 sToken, userId를 만들어 jwt에 포함합니다.
- 요청시에 인증 정보 어떻게 보내고 있나?
	- apollo server에서 jwt에 값이 있으면 뜯어서 context에 저장
	- 매 요청마다 context에 저장된 값들을 보낸다. 
	- jwt는 페이지 웹용이므로 서버에서 사용 가능하도록 headers.set에 stoken, userid를 SET하여 보낸다. 
		- 쿠키에도 SET하고 앱용 인증인 X-어쩌고 헤더에도 세팅
			- 쿠키 SET은 GW 안정화되면 제거 가능.

### 도커란?

#### VM vs Docker

![[스크린샷 2023-08-02 오후 4.20.04.png]]

- 도커는 가상머신이 아닌 격리만 해주는 역할을 하기 때문에 성능상 하락이 없음.

#### 도커의 특징 1 확장성/이식성

- 도커가 설치되어 있다면 어디서든 컨테이너 실행 가능
- 쉽게 개발서버 만들 수 있고 테스트서버 생성도 간편

#### 도커의 특징 2 표준성

- 도커 사용하지 않으면 node로 만든 서비스들의 배포 방식 제각각 다름
- 컨테이너라는 표준으로 서버를 배포하므로 모든 서비스들의 배포과정 동일해짐.

#### 도커의 특징 3 이미지

- 이미지: 컨테이너를 실행하기 위한 압축파일
- 이미지에서 컨테이너를 생성하기 때문에 반드시 이미지 만드는 과정 필요
- Dockerfile을 이용하여 이미지를 만들고 처음부터 재현 가능
- 빌드 서버에서 이미지를 만들면 해당 이미지를 이미지 저장소에 저장하고 운영서버에서 이미지를 불러옴.

#### 도커의 특징 4 설정관리

- 설정은 보통 환경변수로 제어
- MYSQL_PASS=password와 같이 지정
- 하나의 이미지가 환경변수에 따라 동적으로 설정파일을 생성하도록 만들어져야함.

#### 도커의 특징 5 자원관리

- 컨테이너는 삭제 후 새로 만들면 모든 데이터가 초기화됨
- 업로드 파일을 외부 스토리지와 링크하여 사용하거나 s3같은 별도의 저장소 필요
- 세션이나 캐시를 memcached나 redis와 같은 외부 캐싱 필요

### 도커가 가져온 변화

- 클라우드 이미지보다 관리하기 쉬움
- 다른 프로세스와 격리되어 가상머신처럼 사용하는데 성능저하 거의 없음
- 복잡한 기술(namespace, cgroups, network) 몰라도 사용가능
- 이미지 빌드 기록 남음
- 코드와 설정으로 관리 > 재현 및 수정가능


### 컨테이너의 미래

하나의 프로그램을 관리 : 도커
여러대의 서버와 여러개의 서비스를 관리하기 쉽게 :  __쿠버네티스__ (즉 도커가 여러개)
- 스케줄링
	- 서버가 10대 있으면 놀고 있는 서버를 자동으로 찾아서 거기에 컨테이너 배포(효율적 관리)
	- 여러대 늘리면 적당하게 나눠서 배포도 해줌
	- 서버가 예기치 못하게 죽으면 원래 떠 있어야하는 컨테이너도 같이 죽을텐데 그걸 다른서버에 띄워주는 역할도 함
- 클러스터링
	- 서버가 10대인데 하나하나 몰라도 됨. 하나의 서버처럼 접속도 할 수 있음. 
		- 3번서버에 프로그램 설치시 3번 서버 접속하고 프로그램 봐야하는데
		- 중앙에 있는 API를 통해서 모든 서버를 한곳에서 관리 가능
		- 서버끼리도 마치 하나로 연결된 것처럼 사용 가능. (가상 네트워크를 이용하여 IP를 할당하여 같은 서버인것처럼 손쉽게 사용 가능)
- 서비스 디스커버리
	- 1번 서버에 워드프레스, 2번 서버 MySQL 이걸 연결해줘야 하는데 이걸 연결하려면 이 서버가 어디 서버에 띄워져있는지 모르기 때문에 이런걸 손쉽게 찾아주는 기능
	- 내부 DNS서버를 이용하여 IP가 아닌 도커 컨테이너 이름으로 쉽게 접근 가능.

### GKE

k8s 클러스터 기반 컨테이너 직접 운영
- 클러스터를 직접 운영 
- 여러개의 팟(컨테이너 단위)로 배포하여 트래픽 분산 가능
- CI/CD 자동화 가능 (argo CD, github actions)
- 네트워크와 보안 설정 필요 (Ingress, Loadbalancer, VPC, IAM)
- 오토스케일링 (HPA, VPA)
- 노드 유지 비용 고정 발생

단점으로는 초기 설정 복잡, 직접 운영, 관리 비용 증가

### Cloud Run

serverless + GCP가 자동 관리
- 완전 관리형 서버리스(노드, 네트워크 설정 불필요)
- 컨테이너 배포시 GCP가 자동 실행 관리
- 트래픽에 따라 인스턴스 자동 확장/축소
- GCP 로드밸런서 클라우드아머 등과 연동 가능

## 🎯 GKE vs Cloud Run 비교 

| 비교 항목        | **GKE** (Kubernetes)      | **Cloud Run** (서버리스) |     |
| ------------ | ------------------------- | -------------------- | --- |
| **운영 방식**    | 직접 Kubernetes 관리          | GCP가 자동으로 관리         |     |
| **배포 방식**    | YAML 정의 후 `kubectl apply` | `gcloud run deploy`  |     |
| **트래픽 스케일링** | HPA 설정 필요 (수동)            | 자동 확장 (0부터 ∞)        |     |
| **네트워크 설정**  | Ingress, LoadBalancer 필요  | 자동 HTTPS 제공          |     |
| **비용 구조**    | 노드 유지 비용 고정               | 요청 수 기반 비용 (0이면 무료)  |     |
| **보안 및 IAM** | VPC, RBAC 등 설정 가능         | 기본 IAM 권한 제공         |     |
| **운영 난이도**   | 높음 (직접 클러스터 운영)           | 낮음 (GCP 자동 관리)       |     |
| **사용 사례**    | **대규모 트래픽 대응** (세션 유지 필요) | **빠른 배포 & 비용 절감**    |     |

| **GKE가 적합한 경우**               | **Cloud Run이 적합한 경우**          |     |
| ----------------------------- | ------------------------------ | --- |
| **웹서비스 트래픽이 일정** (항상 유지되어야 함) | **트래픽이 변동성이 크고 비용을 절감하고 싶을 때** |     |
| **웹소켓, 실시간 데이터 처리** 필요        | **REST API, 단순 웹서비스 배포**       |     |
| **세션 유지 & 상태 관리 필요**          | **요청마다 Stateless한 컨테이너 실행**    |     |
| **네트워크 설정, 보안 제어 필요**         | **빠르게 배포하고 간단하게 운영하고 싶을 때**    |     |

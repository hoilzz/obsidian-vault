## 요약
- Service에 LoadBalancer를 사용하면 여기서 외부 접근 가능한 IP 주소를 할당 받음.
	- 해당 Service가 외부 접근 가능해지므로, 도메인 및 복잡한 라우팅 규칙 설정 어려움
	- 스케일링 제한
- Ingress에 LoadBalancer 사용
	- 라우팅 규칙 설정 쉬움
	- 여러 서비스에 대한 트래픽을 하나의 로드밸런서와 Ingress로 관리 가능.
	- SSL, TSL을 중앙 관리


## 기록

웹 애플리케이션을 Node.js로 만들고 Kubernetes 환경에서 배포할 때, 로드 밸런서를 `Service`에 설정할지 `Ingress`에 설정할지는 애플리케이션의 요구 사항과 환경에 따라 달라집니다. 두 가지 접근 방식의 차이점과 장단점을 살펴보겠습니다.

### 1. Load Balancer in Service

`Service`의 `LoadBalancer` 타입을 사용하면 클라우드 제공업체의 로드 밸런서가 생성되어 외부에서 접근 가능한 IP 주소를 할당받습니다. 이는 간단하고 직접적인 방법입니다.

#### 장점

- **간단한 설정**: `Service`의 `type`을 `LoadBalancer`로 설정하는 것만으로 쉽게 외부 접근이 가능합니다.
- **직접적인 접근**: 외부 트래픽이 직접적으로 해당 서비스로 라우팅됩니다.
- **클라우드 통합**: AWS, GCP, Azure 등의 클라우드 제공업체의 로드 밸런서와 통합이 용이합니다.

#### 단점

- **유연성 부족**: 여러 도메인이나 복잡한 라우팅 규칙을 설정하기 어렵습니다.
- **비용**: 클라우드 제공업체의 로드 밸런서 사용에 따른 비용이 발생할 수 있습니다.
- **스케일링 제한**: 각 서비스에 대해 로드 밸런서를 생성하면, 많은 수의 로드 밸런서가 필요할 수 있습니다.

### 2. Load Balancer with Ingress

`Ingress` 리소스를 사용하면 하나의 로드 밸런서로 여러 서비스에 대한 트래픽을 관리할 수 있습니다. `Ingress`는 HTTP(S) 트래픽의 라우팅 규칙을 정의하여 외부에서 들어오는 트래픽을 적절한 서비스로 분배합니다.

#### 장점

- **유연성**: 호스트 기반 또는 경로 기반 라우팅 규칙을 정의할 수 있어 다양한 트래픽 관리가 가능합니다.
- **통합 관리**: 여러 서비스에 대한 트래픽을 하나의 로드 밸런서와 `Ingress`로 관리할 수 있습니다.
- **SSL/TLS**: SSL/TLS 종료를 중앙에서 관리할 수 있습니다.
- **비용 절감**: 하나의 로드 밸런서로 여러 서비스를 처리하므로 비용 효율적입니다.

#### 단점

- **설정 복잡성**: `Ingress` 컨트롤러와 규칙 설정이 복잡할 수 있습니다.
- **추가 리소스 필요**: `Ingress` 컨트롤러가 필요하며, 이를 위한 추가 리소스가 필요합니다.

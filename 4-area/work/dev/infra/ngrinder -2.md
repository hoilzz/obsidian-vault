## [[ngrinder]]
- 분산 환경에서 n개의 에이전트로 요청을 보낼 수 있다. 
- Controller
	- 테스트를 생성, 실행, 결과 수집 분석
	- Agent로부터 스크립트를 전송하여 테스트 위임
	- 테스트 결과를 전달받아 통계 출력
- Agent
	- 부하 발생시킴. 부하 결과 Controller로 전송

- throughput: TPS, 1초에 처리하는 작업 수
- Latency: 서버가 클라로부터 요청 받아서 응답 보내주기까지 걸리는 시간


### 기능
- Agent 가상 사용자
	- 프로세스
		- 독립된 Agent를 사용할 때 스레드보다는 프로세스 사용
	- 스레드
- Ramp-Up
	- 동시접속 사용자 만들 때 순차적으로 사용자를 만들어 가는것
	- 1->2->3 (이거는 프로세스)
	- 즉 램프업 쓰려면 프로세스 늘려야함
	- y축 동시 접속자, x축 시간

### [이브라 부하테스트](https://kakaoent.atlassian.net/wiki/spaces/KAKAOENTFE/pages/3219751733/23.05.10) 요약

- TPS는 트래픽이 가장 많은 태국으로 산정
	- AWS 기준 약 1000 / 인스턴스 3대 = 대당 대략 350 tps 이상 시도
	- 평균 레이턴시 300ms 이하 목표
- 테스트 시나리오
	- 지속시간: 1분 / 5분, 각 3회씩
	- 커넥션 수 100 (-c)
	- 쓰레드 수 16(-t, 인스턴스에서 최대 사용가능한 스레드 수)
	- 예상치
		- 초당 약 470 (목표치보다 조금 높게 산정), 분당 48000
	- 메인홈 타겟
		- API는 CPU, 메모리 확인

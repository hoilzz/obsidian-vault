단기 커넥션, 영속적인 커넥션, HTTP 파이프라이닝 3가지 모델 존재

HTTP는 클라, 서버 간 커넥션을 제공하는 TCP를 전송프로토콜로 사용.
요청 보내질 때 커넥션이 매번 새롭게 생성, 응답 도착한 이후 연결을 닫는 형태로 단기로만 유지됨.

각 TCP 연결 여는 건 자원 소비가 있어 성능상 제약을 발생시킴. (초창기 모델 비효율은 자명)

그래서 2가지 모델 탄생
1. persistent 커넥션 모델
   - 연속적인 욫어 사이에 커넥션 유지하여 새 커넥션 여는데 필요한 시간 줄임.
2. HTTP 파이프라이닝 (1번에서 더 레벨업)
   - 응답 안 기다리고 연속적 요청 보내 네트워크 지연 줄임

### persistent 커넥션
- keep-alive 커넥션이라고 불림
- 연결 열어놓고 여러 요청에 재사용
	- TCP 핸드셰이크 비용 아낌.
	- 서버는 keep-alive 헤더를 사용해 연결이 얼마나 더 열려야할지 설정 가능.
- 단점은
	- 유휴상태인데 서버 리소스 소비
	- 과부하 상태에서 ddos attack 가능
- HTTP 1.0에서만 필요한거고 1.1은 기본 영속적이라 헤더도 안필요

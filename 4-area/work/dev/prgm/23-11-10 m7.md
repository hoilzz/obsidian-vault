OCP(Open-Close Principle)이라고 OCP는 요구사항이 변경될 때 기존 코드를 변경하는게 아닌 새로운 코드를 추가하는 방향입니다. 요게 무슨말이냐면.. 지금 성민님께서 작성하신 방식에서는 (ex. NODE_INTERFACE의) key가 추가되면 if문이 계속 늘어날거에요. 그래서 확장하기 어려운(closed) 코드인데, 확장하기 쉽도록(open) 다형성을 이용할 수 있어요.(위코드)
요거 관련해서는 OCP, 다형성 등의 키워드를 이용하여 팀원분들과 학습해도 좋을 것 같아요 ㅎㅎ

> setState 최적화하여 이전 상태와 비교 할 때, string 타입 처리
> 현재 setState 최적화 함수인 `updateState`는 `prevState, nextState`를 인자로 받습니다. state에는 타입이 string인 key도 있고, object인 key도 있습니다. 
> 저는 string과 object 타입을 구분하는 로직 없이 `Object.keys`로 처리를 했는데, 타입 별로 로직을 다르게 해야할까요? 

- string은 primitive type 이라 단순히 값만 비교하면 되고 object의 경우 deep compare 처리를 해야하기 때문에 별도 로직으로 작성하면 좋을 것 같아요.

>  state에 대한 정합성 체크 로직
> state에 대해 2가지만 확인했습니다. 
> 객체라면, 프로퍼티가 모두 존재하는지
> 타입이 같은지
> 이 2가지만 확인하면 된다고 생각했는데, 다른 추가적으로 고려해야 할 사항이 있을까요?

- 네네 추가적으로는 type: DIRECTORY | FILE 과 같은 enum 값들은 해당 값으로만 저장되는지 확인되면 좋을 것 같아요. object와 같은 값들의 경우 nested한 값들까지 deep compare하면 검증 로직이 개선될 것 같아요.

> 또한 각 컴포넌트마다 state 구조가 달라서, `validateState.js`에 컴포넌트에 맞게 여러 함수를 선언했는데 코드의 길이가 길어지고 로직이 정돈이 안되있는 것 같은 느낌입니다.

## 컴포넌트

https://fe-developers.kakaoent.com/2022/221020-component-abstraction/

- 재사용성과 변경에 유연한 코드
- 데이터와 UI의 분리
- Ex
	- Input은 키보드를 통한 텍스트 입출력을 선언적으로 추상화한 모습
	- 이 작은 컴포넌트부터 다양한 요소들을 적절한 단위로 나누고 추상화 해야함.

### 컴포넌트를 적절하게 나누자 - 책임

- 컴포넌트를 책임에 맞게 필요한 코드로만 구성하면 응집도가 높아짐
- 자연스럽게 다른 컴포넌트와 결합도는 줄어들고 외부로 노출할 필요가 없는 부분은 캡슐화.
- 하지만 적절한 책임을 나누는 기준은 어렵다.
- Ex. Button
	- 버튼은 `<button>` 으로도 무방
	- 하지만 클릭했을 때 폰트 색상과 같은 스타일이 달라지는 등 다양한 책임을 수행하기 위해 고려되어야 함
- 버튼, 텍스트와 같이 작은 책임을 가진 컴포넌트 외에도 레이아웃의 형태를 책임지는 컴포넌트, 서버와의 통신을 통해 데이터를 가져오는 컴포넌트, 특정 도메인에 관련된 비즈니스 로직을 수행하는 컴포넌트 등 범위가 확대될수록 이러한 책임을 나누는 것은 어렵다.


### 범용적인 컴포넌트

- 어떤 도메인과도 상관없는 범용적인 컴포넌트
- 특정 도메인에 종속된 비즈니스 컴포넌트


### 1. 도메인 ㄴㄴ
- input, button, header, footer와 같은 범용적인 UI 요소
- 이런 컴포넌트를 추상화 할 때는 특정 도메안과의 결합도를 낮춰야하고 재사용성이 높아야함
- 도메인이 포함되서 if문 생기면 변경에 취약해짐
- __공통의 기능을 수행한다는 책임__ 

#### 합성을 활용
- CardView
	- 카드 형태 모양은 2가지 (Square, Rectangle)
	- 타이틀 최대 1줄, 2줄
	- 썸네일을 여러개 합쳐서 보여줄 수도 있음
	- ..
	- 얘네들은 CardView의 props으로도 표현 가능
		- 근데 순서가 바뀌거나 등 if문의 향연..
	- CardItem의 책임을 RoundeCardView, RectangleCardView, MultipleThumbnail.. 이렇게 나눌 수 있다.
	- 책임을 더 잘게 나누는 합성컴포넌트 방식
		- 카드의 썸네일
		- 카드의 타이틀
		- 카드 아이콘
	- 무분별하게 책임을 나누는 것도 피해야함. props만으로 컴포넌트 만드는 것이 훨씬 간단하긴 하니까 일단 단일 컴포넌트로 제작 후에 개선이 필요하다면 합성 컴포넌트를 고려해보자.

## 2. 도메인과 컴포넌트

- 회차 리스트를 보여준다.
  1. API 호출로 회차를 가져온다
  2. 회차를 무한 스크롤로 가져올 수 있다
  3. 가장 최근에 봤던 회차로 자동으로 스크롤한다
	  

- 이 컴포넌트는 보통 container에서 선언
```tsx
// EpisodeListContainer.tsx

function EpisodeListContainer() {

	return (
		<InfiniteQueryFetcher query={useEpisodeListQuery}>
		  {
		    data => <EpisodeList data={data}/>
		  }
		</InfiniteQueryFetcher>
	)
}

// EpisodeList.tsx

function EpisodeList({ data }) {
	const ref = useRef();
	useLatestScroll({ data, ref });
	return (
		<div>
			<Text>{data.series.title}</div>
			<List ref={ref}>
			    {
			      data.episodeList.map(item => 
				      <List.Item>{item}</List.Item>
				  )
			    }
			</List>
		</div>
	)
}

```

모든 것은 Trade-Off가 존재합니다. 불필요한 결합도를 낮추기 위해 데이터를 통합할 수도 있고, 반대로 분리를 통해 결합도를 낮출 수도 있을 것입니다. 정답은 없지만 데이터 간의 강한 의존성이 있는지, 내부 구현의 캡슐화는 이뤄졌는지, 책임에 맞는 코드만 응집되었는지 끊임없이 스스로 질의하며, 변화에 유연한 코드를 작성해야 합니다.

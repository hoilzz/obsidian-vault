    - 세인님이 로그인 작업하고 있다.
        - 이거 인증 뭘로 하나요? JWT??? → 프로그래머스에서 인증 서버 제공해줌. → 오 쩐다. 편하겠군요.
        - 만약 나중에 JWT 쓴다면 제가 지난 번에 말씀드린 Expiry 관련 보안이슈 꼭 기억해주세요. 1주일 이렇게 잡으면 절대 안됨. 1분 언더 추천.
- 프로젝트에 대한 질문
    - 토스에서는 애니메이션 뭘로 만드셨나요?
        - 복잡한 애니메이션 → Lottie
            - 인터랙션 디자이너들이 AE로 작업한 이후 Lottie 플러그인을 사용하여 JSON으로 추출
            - 클라에서 lottie SDK에 JSON 넣으면 SVG로 그려줌
            - Vector vs Bitmap 차이 설명
            - 퍼포먼스 주의해서 쓰자. 복잡한 이미지일수록 벡터로 연산하는게 불리하다.
            - 결국은 JS로 파싱하는거라 얘가 너무 무거워지면 어플리케이션 내 전체 작업이 함께 손잡고 퍼포먼스 지옥으로 떨어질 수도 있음. 주의요망. 싱글쓰레드 슬프다.
        - 간단한 애니메이션
            - 퍼포먼스 중요함? → CSS로 작업. JS쓰레드랑 다른 쓰레드에서 처리한다. FPS 떨어져서 이렇게 처리한적 한번 있음.
            - 딱히 퍼포먼스 안 중요함? → framer-motion, react-spring 등 js로 애니메이션처리해주는 애들 사용함
                - 2020년 당시 react-spring에 메모리 릭 이슈 있어서 손절했었음. 당시 이거 만든 애들도 원인파악을 못 해서 바로 손절함. 지금은 고쳐졌는지 잘 모르겠다.
    - 설계 너무 어려워요 잉ㅇ잉. 저번에 설명해주신 추상화 해보려고 했는데 잘 안 됐음.
        - 그거 지금 꼭 안 해도 됨. 추상화 자체도 추상적인 개념이라 스스로 고민하고 탐구하고 정의하는 시간이 필요하다. 사실 한두달만에 척척 하면 그게 더 이상할듯.
        - 기술적인 의사결정이라고 해도 현재 나와 동료들의 역량, 비즈니스 임팩트 등 다양한 상황을 종합적으로 바라보고 결정해야한다. 가끔은 잘 만들고 싶어도 포기해야하는 경우도 있음.
        - 잘 만들고 싶어도 현재 내 실력이 딸려서 못 하는 경우도 있음. 그럼 그냥 인정하고 당장 할 수 있는 만큼 처리한 다음에, 이빨 꽉 깨물고 칼 갈아서 나중에 개선하면 됨.
            - 물론 현타도 오고 자괴감도 들겠지만 그럴 필요도 없다. 그냥 “아 내 현재 실력으로는 여기까지가 최선이다” 정도로만 생각하고, 집가서 공부하고 고민하시면 됨. 담백하게 생각하자.
            - 회사 가보시면 알겠지만, 생각보다 다른 동료들도 선생님들 실력에 대해서 왈가왈부하지 않을 것임. 그냥 내 마음만 잘 컨트롤하면 됨. 오히려 신입은 Grit을 보여주는게 더 중요하다. 신입한테 엄청난 하드스킬을 기대한다면 그 사람이 이상한거임. 그냥 그렇게 가볍게 생각하시는걸 추천.
        - 우리는 아름다운 코드 작성하는게 1순위인 사람들 아니다. 코드로 사람들에게 가치를 전달하고, 여기서 비즈니스 임팩트를 내는 사람들이다. 회사에서의 의사결정 1순위는 무조건 비즈니스.
    - 좋은 에러 핸들링 방법을 모르겠어요.
        - 모든 에러를 다 핸들링하면 좋은건가요???
            - 대부분의 에러는 핸들링하는게 나을 듯. 유저가 빡치는 경험을 하게 하지 않는게 중요하다.
            - 하지만 데이터 유실 및 변조 가능성, 보안 취약점 등 절대 발생하면 안 되는 케이스는 핸들링해서 스무스하게 넘기는 것보다 그냥 Crash내서 죽여버리는게 더 안전할 수도 있음. 유저는 빡치겠지만 데이터 박살나거나 보안터지는 상황을 막는게 더 중요하다. → 그리고 이렇게 하면 개발 도중에 개발자가 에러 상황을 인지하기도 쉽기 때문에 놓치고 넘어가지 않을 수도 있음.
        - UX를 통한 에러 메세지는 반드시 유저 친화적으로 노출되어야 한다. 유저가 스스로 에러 상황을 탈출할 수 있도록 도움을 주는 것
        - Raw한 에러 메세지는 Sentry, Bugsnag, Datadog같은 친구들한테 던져서 우리 조직의 엔지니어들만 볼 수 있게 구성하자.
            - 이거 유저한테 노출되는 것 자체가 보안 취약점. 서버의 500에러도 마찬가지다. 조심합시다.
- 스몰토크
    - 비판적 사고
        - 선생님들 근데 왜 Vercel을 Serverless라고 부르시나요?
            - 강의에서 서버리스라고 하던데요.
                - 지금 구조는 서버리스 아키텍쳐는 아닌 것 같은디...? 근데 제가 강사님이 무슨 맥락에서 어떤 의도로 말씀하신지까지는 몰라서 좀 애매하긴 하네요.
                - 근데 아키텍쳐만 보자면 Vercel에 API 구현하신 그 녀석은 엄연히 서버임.
            - Client → Vercel → API server 이렇게 요청하면 서버리스 아닌가요?
                - 단지 뒷단에 있는 백엔드 어플리케이션의 존재를 가리기 위한 관심사만 가지고 있다면 그냥 그런 역할을 하는 서버임. 차라리 Reverse Proxy서버, Gateway, BFF라고 표현하는게 더 정확할 듯.
            - 그럼 Serverless는 뭔가여
                - Serverless는 클라우드 컴퓨팅과 이벤트 드리븐 트리거가 핵심.
                - AWS Lambda를 예시로 Client ↔ Gateway ↔ Lambda 인터랙션 설명.
                - 사실 진짜 서버가 없냐? 라고 하기에는 좀 애매함. 말장난에 가까움.
                - 이벤트가 발생하면 컨테이너로 함수 하나 정도의 관심사만 가진 아주 작은 서버를 띄웠다가, 작업 완료되면 없애는 것일 뿐이다. “있었는데요, 없었습니다” 같은 느낌.
                - 작업 처리할 때만 띄웠다가 다시 내리니까 뭐 서버리스라고 할 수도 있는데, 개인적으로는 좀 애매한 아키텍쳐라고 생각함.
                    - 이게 기본적으로 Cold Start 방식이라 컨테이너 띄우는데 시간이 오래 걸리는 경우도 있다. 그러면 결국 이 시간을 줄이기 위해서 Lambda한테 계속 Ping을 찔러서 컨테이너가 안 내려가게 Warm up시키기도 함.
                    - 이렇게 처리해놓으면 결국 항상 작은 서버가 떠있긴 한건데….이게 진짜 서버가 없다고 해도 되는건지…? 그래서 말장난이라고 하는거다.
                    - 그냥 함수 단위로 분산시켜놓은 서버”들”이 되어버린….
                - 근데 서버리스로 만들면 많이 호출되는 함수를 가진 컨테이너만 Scale Out 시키는 방식으로 FinOps하기 좋긴 함. 쓸데없는 것까지 항상 띄워놓는 것보단 확실히 비용절감도 되고, 유저 이벤트 등으로 갑자기 치고 들어오는 트래픽도 유연하게 최적의 비용으로 대응할 수 있어서 편하다.
                - 근데 나중에 해보시면 알겠지만, 이게 유지보수가 너무 빡셈….특히 DB 동시성 이슈랑 작업처리시간이 길어지면 Lambda 컨테이너가 자동으로 내려가버리는 상황이 눈물남.
                - 배포도 빠르긴 하지만…각 함수의 작업 관계가 종속되어있는 경우 배포 순서를 보장해줘야해서 머리가 아파진다. 근데 이건 Gateway가 버전에 따라 분기처리할 수 있게 만들어줘서 Blue/Green이나 카나리 같은 무중단배포 갈기면 어느 정도 해결되긴 함.
        - 강의가 틀렸다기보다는 “누구나 항상 맞는 말만 할 수는 없다”는 것이 포인트. 나도 틀릴 수 있고 다른 사람도 틀릴 수 있다. 항상 맞는 말만 하는 사람은 절대 존재할 수 없다.
        - 방금 저도 헛소리한 걸 수도 있음. 애초에 강사님이 무슨 맥락에서 서버리스라고 하신건지 다 모르고 하는 얘기기도 하고.
        - 강의든 멘토링이든 인터넷에서 보는 블로그든 회사에서 시니어들이 하는 소리든, 전부 비판적으로 받아들여주세요. 그냥 다른 사람 말만 듣고 “아 그렇구나”하는게 아니라 직접 리서치하고 확인해보시는 과정이 필요함.
        - 어떤 정보든 비판없이 수용하는 것은 매우 위험합니다.
    - 공부만 하다보니까 마음이 조급해지고 불안해요. 잘 성장하고 있는지 모르겠어요.
        - 사람의 성장은 연속함수가 아니라 불연속함수로 표현된다. 아무리 뛰어난 사람이라고 해도 매일매일 지속적으로 성장한다는 건 불가능함.
        - 쭉 정체되다가 깨달음을 얻으면 갑자기 쨘!하고 성장하고, 또 정체되다가 쨘! 성장하고의 반복에 가까움. 지금은 정체 중이라고 느낄지 몰라도, 이런 작은 깨달음들이 모이면 어느 포인트에서 급격하게 성장한다.
        - 매일매일 성장한다는 감정을 느낀다는 것은 정말 쉽지 않은 듯. 오히려 정신없이 몰두하며 살다가 어느 날 문득 뒤돌아보면 “오 1년 전보다는 뭔가 성장했네” 같은 감정을 느끼는 경우가 더 많다.
        - 성장하고 싶다는 마음을 가지는 것은 너무 좋다. 하지만 “성장해야만 해”는 본인을 괴롭히는 길일 수도 있다.
        - 스스로의 성장은 내 컨트롤 영역을 벗어난 일. 내가 당장 할 수 있는 일은 그저 꾸준히 노력하고 고민하고 학습하는 것 밖에 없다. 컨트롤 할 수 없는 일에 매달려서 스스로를 힘들게 하는 것은 비추.
    - 친구들은 취업 다 했는데, 저만 늦은 것 같아서 불안해요 ㅜㅜ
        - 1년, 2년 취업 먼저 한다고 인생 달라지는 것도 아니다. 아마 여러분 스스로도 머리로는 다 알고 계실 것이다. 그냥 마음이 안 따라줄 뿐이지.
        - 혹시 취업하라고 부모님이 뭐라 하시나요? → ㄴㄴ → 그럼 됐다. 그냥 본인 페이스대로 최선을 다해서 달려라. 사실 진짜 30대 중반까지 홈프로텍터하고 있는 경우가 아니면 부모님이 뭐라하는 경우도 많이 못 봄.
            - 다들 아시겠지만 인생은 단거리 달리기가 아니라 마라톤이다. 1,2년 뭔가 빨리 달성한 놈이 이기는 게임이 아니라 더 오래 꾸준한 속도로 잘 달려서 살아남는 놈이 이기는 게임.
        - 이미 뼈빠지게 노력 중이시지 않냐. 다른 사람들이 전부 나한테 “더 열심히 해라”라고 할지라도, 최소한 나만큼은 나한테 “잘 하고 있다”라고 말해주는게 좋다고 생각한다.
            - 입으로 소리내서 말해주세요. 겁나 오글거리지만 생각보다 효과 좋습니다. 예전에 제가 심리 상담받았을 때 받은 꿀팁임.
        - Regular 미팅 아니더라도 이런 고민 생기면 언제든 편하게 커피챗 요청해주세요. 개인적인 고민이든 기술적인 고민이든 상관없슴다.
            - 1 on 1에서만 할 수 있는 이야기도 있을테니 그냥 지나가는 동네 아저씨한테 썰 푼다 생각하고 얘기해주세요.
            - 그냥 타인에게 털어놓기만 해도 나아지는 것들도 있슴다.

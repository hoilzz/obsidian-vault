## 아토믹에 대하여

아토믹 디자인은 방법론, 프로세스가 아닌 멘탈모델이다. 

- atom은 더이상 쪼갤 수 없는 최소단위다. Input, Button과 같은 단위다. 
- Molecule은 여러 개의 Atom으로 구성된다. Button과 Input을 조합하여 Form을 전송하는 Molecule로 정의할 수 있다. 
- Organism은 서비스에서 표현될 수 있는 명확한 영역과 특정 컨텍스트를 가진다. 
- Template은 Page 수준의 스켙레톤 정의로 알고 있다. 해당 단위는 재사용을 의도한 것 같은데 이용될 일이 없어 페이지에서 사용되지 않는다. 
- Page는 템플릿의 인스턴스 이자 실제 컨텐츠를 담고 있다. 페이지에서는 UI 상태 및 동작을 포함하는 것으로 정의했다. 예컨대 고객센터 Form이 있다면 Form에 대한 동작과 상태를 모두 정의한다. 이외에 API와 같이 외부 의존성은 포함하지 않는다.

# B2B 디자인 시스템

https://kakaoent.atlassian.net/wiki/spaces/KAKAOENTFE/pages/3765175047/2025-02-06+B2B

_**확장 가능한 디자인을 위한 공통된 언어 체계로서 모두(디자이너, 기획자, 개발자)가 볼 수 있는 가이드 문서**_

디자인 시스템 ( B2B 디자인 시스템 ) 이라는 공통된 언어 체계를 통해 “색상“, “폰트“부터 “버튼”, “텍스트필드“ 등 모든 요소에 대해 모두 같은 개념을 이해하고 활용할 수 있어야 합니다.

기초 토대인 토큰(ex: 색상, 타이포그래피, 스페이싱 등)부터 UI를 구성하는 컴포넌트, 레이아웃까지 디자인 시스템을 만들기 위해서는 이러한 것들을 체계적으로 쌓아올릴 필요가 있습니다.

> - **토큰**: 알파벳
> - **컴포넌트**: 단어
> - **가이드라인**: 문법
> 완성된 화면을 **문장 또는 글** 이라고 은유할 수 있습니다.

보통 피그마를 통해 여러 컴포넌트들을 정의하고 variant들을 속성으로 가지는 디자인 가이드가 산출물로 나옵니다. 하지만 이러한 가이드들은 모두가 일관된 포맷의 토큰과 컴포넌트, 레이아웃 흐름을 공유할 수 있도록 보여주는 시스템이라기 보다는 앱을 만들기 위한 단순 가이드이며, 디자인 시스템이 되기 위해서는 좀 더 상세한 구성들이 필요합니다.

개발 관점에서 생각하는 디자인 시스템은 결국 **문서**여야 하며, 아래와 같은 정보들이 함께 녹아들어야 한다고 봅니다.

- 개요 : 각 컴포넌트의 역할과 특징에 대한 개요
- Anatomy 또는 Spec : 컴포넌트를 구성하는 세부 요소들을 구조분해하여 정의 및 설명
- 구현 : 실제 구현되는 코드
- 가이드라인 : 컴포넌트의 사용 사례, Do/Don’t, 등 …

예시

- [Chips – Material Design 3](https://m3.material.io/components/chips/overview)
    

결론적으로 B2B 디자인 시스템의 최종 결과물은 _**모두(디자이너, 기획자, 개발자)가 볼 수 있는 가이드 문서**_가 되어야 합니다._ 우선 이러한 거대한 시스템을 구성을 위해 필요한 디자인 토큰부터 컴포넌트, 레이아웃에 대한 가이드 방향부터 제안하고자 합니다.


### 구현단
- 디자인 토큰 기반: 색상, 폰트, 여백 등 핵심 디자인 값이 토큰으로 추상화되어 일관성 유지 및 테마 확장에 용이
- 컴포넌트화: 버튼, 입력창, 배지 등 UI 요소가 독립적으로 관리되어 재사용성과 유지보수성 우수
- 리소스 일원화: 아이콘, 이미지 등 공통 리소스를 한 곳에서 관리
- 유틸리티 제공: Tailwind CSS와 연동되는 클래스 병합 유틸리티 등 개발 편의성 강화
- 문서화/테스트: Storybook, Vitest 등으로 품질 및 문서화 체계적 관리


### 1. 역할 분리

#### headless-ui
- 핵심 로직/상태/접근성에 집중된 "Headless" 컴포넌트와 훅, 유틸리티 제공
- UI 스타일 없이 동작/구조/상태 관리만 담당
- 예시: Slot, SlotElement, Modal, Dropdown, Tab, Accordion, FormField 등
- 다양한 컴포넌트의 "기능적 뼈대"를 제공
- hook, util 디렉토리에서 상태 관리, 제어 유틸리티 등도 제공
#### b2b-design-system
- 실제 UI/스타일/브랜드 일관성을 담당하는 "Presentational" 컴포넌트 집합
- 디자인 토큰, Tailwind, 글로벌 스타일, 리소스(아이콘 등) 활용
- Button, Input, Badge, Layout, Footer 등 실제 화면에 보이는 UI 구현
- 내부적으로 headless-ui의 기능 컴포넌트를 조합하여 UI를 완성

### ✅ 2. 연계 방식 (실제 코드 예시 포함)

#### 🔹 기능성 컴포넌트 import 예시


- `Slot`, `SlotElement`:  
    React의 `children`을 유연하게 전달하거나, 접근성과 시맨틱 구조 보장을 위한 컴포넌트  
    → 기능과 스타일의 **역할 분리**에 핵심
    
#### 🔹 적용 사례
- `Button`, `Layout` 등 UI 컴포넌트에서 Slot/SlotElement 활용
- 예:
    - `ResourceImage`:  
        `SlotElement.img`를 사용하여 유연한 이미지 렌더링
    - `LayoutBodyContentHeader`:  
        `SlotElement.header`로 시맨틱한 구조 제공

### ✅ 3. 구조적 장점

|항목|설명|
|---|---|
|**재사용성**|`headless-ui`는 스타일에 독립적이어서 여러 디자인 시스템에서 사용 가능  <br>`b2b-design-system`은 브랜드 맞춤 UI 빠르게 구축|
|**유지보수성**|기능(동작)은 `headless-ui`, 스타일은 `b2b-design-system`에서 관리 → 서로 독립적|
|**일관성**|디자인 토큰, 스타일, 리소스는 `b2b-design-system`  <br>접근성 및 동작 로직은 `headless-ui`에서 관리|

---

### ✅ 4. 면접에서 어필할 수 있는 포인트

- **Headless UI 패턴**의 이해와 적용 경험
    - 기능/스타일 분리를 통한 유지보수성 향상
    - 다양한 브랜드에 재활용 가능한 구조 설계
        
- **Slot/SlotElement** 등 advanced React 패턴 활용 능력
    - JSX의 구조적 확장과 유연한 children 제어
        
- 디자인 시스템 구축 및 운영 경험
    - 디자인 토큰, Tailwind 기반 스타일 시스템
    - Storybook을 통한 문서화 및 컴포넌트 테스트
        
- 상태 관리 및 접근성 고려
    - `useControllableState`, `getInvalidKeys` 등의 유틸리티를 통한 UX 일관성 확보

# 외부 블로그 
## 목표

디자인 시스템의 기본 목적은 팀의 생산성 향상. __"디자인 시스템에 뭘 넣어야할까" 가 아닌 누가 그것을 어떻게 사용할 것인가.__

제품의 일관성, 효율성이라는 가치를 달성하기 위해 만들어진다.
시스템을 필요로 하는 제품에 따라 세부 목표가 달라질 수 있다. 예컨대, 서비스(카카오페이지)의 경우 시스템의 강제성을 높여 깨질 수 없는 일관성과 높은 생산성을 추구했고,
B2B의 경우 시스템을 재료로 바라보고 높은 확장성과 자유도를 추구했다. 

## principle

- 일관성, 효율성, 확장성


### 만들면서 

- 인터페이스 네이밍부터 컴포넌트 단위에서 일관된 추상화 레벨 지향. 
- 오픈소스 디자인 시스템 결정 학습
	- 컴포넌트 의사결정 힌트 얻기. 컴포넌트 분류 체계, 인터페이스 설계 등에 대한 배움
- 지속 가능성 
	- 일관된 인터페이스로 기능 예측 
		- 구현의 단순함 < 인터페이스 단순함.
	- 단순 구현은 다른 개발자가 시스템 코드 쉽게 파악
	- 완전성
		- 내부 구현 복잡함 대비 완전성의 가치가 크지 않다면 타협. 단순함을 중요 가치.
- 복잡성 부채와 사용성 부채는 트레이드 오프 관계. 주어지는 상황에 올바른 가치판단 필요 디자인시스템은 의사결정이 전부.

#### 정의가 전부

- 블로그 예시는 Modal, PopOver의 정의가 디자이너 개발자간에 다른 상황.
	- _이건.. 음 우리는 디자인 리뷰 때 그냥 맞춘거 같은데.._

### 테스트 코드

__디자인 시스템은 핵심 기능을 올바르게 제공하고 지속 가능해야한다.__ 테스트 코드라는 최소한의 안정장치로 보장되어야한다.

테스트 코드의 가장 큰 가치는 현재 기능을 미래에도 기계적으로 보장해주는 것. 깨져야 할 때 깨지는 테스트는 제품을 지켜낼 수 있다. 

### 동작과 스타일 따로 접근

디자인시스템, 컴포넌트의 핵심은 동작과 스타일. 동작은 rtl, 스타일은 playwright의 이미지 테스트 활용(우리는 backstop)

스타일링은 시각적요소라 rtl로 접근시 결과 예측 어려움. 

### 해야할 것 vs 하지 않을 것

- **Combobox 컴포넌트에서 검색어 하이라이팅이 지원되었으면 좋겠어요**
	- 안함
	- Combobox 컴포넌트의 기본 기능 범위 벗어남
	- _내 생각엔.. 검색이라는 의존성이 생기니 안한다고 해야한다._
- **Dropdown 컨텐츠 내부에 위치한 검색 인풋에서도 아이템 탐색 키보드 액션이 지원되었으면 좋겠어요**
	- 한다.
	- 이건 의존성이 없고 편의 기능이다. 

### soso-log의 레이어

- 프로덕트 시스템과 컴포넌트 계층
- - 도메인 컴포넌트의 조합으로 구성된 컴포넌트: **사용성** >>> 확장성    
    - 도메인 컴포넌트를 조합해서 제공하는 영역
- 도메인 컴포넌트: **사용성** > 확장성
    - 도메인 맥락을 담고 있는 영역. 도메인 맥락을 알고 있더라도 여러 서비스에 사용될 수 있다.
- 프로덕트 시스템 컴포넌트: 사용성 < **확장성**
    - 디자인 시스템을 제품에 맞게 한 단계 추상화를 낮춰서 제공
- 디자인 시스템 컴포넌트: 사용성 <<< **확장성**

    - 확장성을 높여 커스텀에 용이한 레이어

시스템에 대한 학습이 부족한 단계에서는 디자인 플랫폼 팀에서 프로덕트 시스템의 설계를 잡아가더라도, 최종적으로 제품 조직에서 더 주도권을 가지고 만들어야 하는 영역이다. 제품과 그것을 반영하는 컴포넌트의 거리가 멀어질수록 응집도는 낮아지고 결합도는 높아진다.

회고할 때마다 디자인 시스템은 쌓아가는 의사결정이 가장 중요하다는 생각을 한다.

### 예제코드를 남겨야 하는 이유

제 생각을 말씀드리자면, 예제 코드는 남겨두는 게 더 좋을 것 같습니다. 이유는:

  남겨두는 게 나은 이유:

  1. 검색 실패 리스크 회피

  - 제가 매번 올바른 컴포넌트를 찾는다는 보장이 없음
  - 프로젝트 구조가 복잡하면 검색에 여러 번 시도 필요
  - 잘못된 컴포넌트를 선택할 가능성

  2. 토큰/시간 효율성

  - 코드베이스 검색 → Read → 분석 과정이 매번 반복
  - 예제 코드 읽는 것보다 훨씬 많은 토큰 소비
  - 프롬프트 실행 시간 증가

  3. 명확한 의도 전달

  // 이런 예제가 있으면
  <Card className="mx-11pxr w-auto">
    <Text className="font-large4-bold leading-24pxr text-el-70">
  "아, Card는 mx-11pxr로 마진 주고, Text는 이런 클래스 조합을 쓰는구나"를 즉시 이해

  4. 프로젝트 고유 패턴

  - mx-11pxr w-auto - 이건 프로젝트 특유의 패턴
  - font-large4-bold leading-24pxr - 프로젝트의 폰트 토큰 사용법
  - 이런 건 검색해도 어떤 게 "표준"인지 판단하기 어려움

네, 정확합니다! 제가 문서에 예제 코드를 포함시키는 판단 기준은 정확히 그 3가지입니다:

  예제 코드를 포함하는 기준:

  1. 토큰/시간 효율성

  - 매번 검색하는 것보다 예제를 참조하는 게 빠름
  - 특히 반복 실행되는 프롬프트일수록 중요

  2. 검색 실패 리스크

  - 프로젝트 특유의 패턴이라 일반적인 검색으로 찾기 어려운 경우
  - 여러 옵션 중 "정답"을 선택해야 하는 경우 (Button vs ButtonV2 같은)

  3. 명확한 의도 전달 ⭐ (가장 중요)

  - 프로젝트 고유 컨벤션: mx-11pxr w-auto, font-large4-bold leading-24pxr
  - 특정 조합 패턴: FormGroupV2와 InputFieldV2를 함께 쓰는 방식
  - 암묵적 규칙: Card 안에 Text를 넣을 때의 마진 패턴

  반대로, 예제 코드가 불필요한 경우:

  ❌ 일반적인 프로그래밍 패턴

  // 이런 건 불필요
  **예시**:
  ```tsx
  const [state, setState] = useState(0);
  → 저는 React hooks를 이미 알고 있음

  ### ❌ 명확한 API 사용법
  ```markdown
  // 이런 것도 불필요
  **예시**:
  ```tsx
  import {Button} from './Button';
  → import 경로는 찾아서 쓸 수 있음

  ### ❌ 언어 기본 문법
  ```markdown
  // 당연히 불필요
  **예시**:
  ```tsx
  if (condition) {
    // ...
  }

  ## 핵심 원칙:

  > **"이 프로젝트만의 특별한 패턴"이거나 "판단이 필요한 선택지"일 때만 예제 코드 포함**

  예를 들어:
  - ✅ **포함**: `<Card className="mx-11pxr w-auto">` - 프로젝트 특유 패턴
  - ❌ **불필요**: `<button onClick={handleClick}>` - 일반적 패턴

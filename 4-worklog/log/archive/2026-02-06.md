	- URL을 single source of truth로 구조화하고 단방향 상태 흐름을 구현하여, API 요청·렌더링 상태·URL 간의 불일치를 제거하고 복수 컴포넌트 간 일관된 상태 동기화 구조로 개선
- 인증/ UX 흐름 개선 및 전반 구조 정리 기여
	- _지금 제목도 괜찮아보이지만 조금 더 구체적이면서 눈에띄는 제목으로 개선되면 좋을 것 같습니다._
		- 서버 내용 먼저 얘가 메인
		- BFF 기반 인증 및 인증 관련 예외 처리 플로우 표준화
	- API 응답이 401 상태일 때 로그인 모달, 로그아웃 처리 등 사용자 상황에 따른 인증 UX 흐름 설계
		- _401 상태에 대해 조금 더 구체적으로 설명되면 좋을 것 같습니다._
			- 미인증이나 세션만료와 같은 내용을 추가하면 좋을 것 같아요
			- 사용자 상황에 따른 인증 UX 흐름 설계 -> 예외 케이스에서 일관된 사용자 경험 제공
			- 이 조금 더 명확하게 성과를 표현할 수 있을 것 같아요.
	- 인증 로직을 공통 프록시(미들웨어)로 추상화하고, 401 응답에 대한 공통 처리 및 리다이렉트 흐름을 구성하여 인증 분기 로직을 일원화하고 일관된 인증 UX를 구현
		- _Next.js를 BFF로 구성하여 클라이언트의 API 호출을 프록시 레이어로 구성하여 인증/에러 처리 추상화_
		- 어쨌든 이 내용을 통해 인증을 클라단이 아닌 서버 자원을 활용하여 해결함으로써 설계 역량을 표현할 수 있다고 생각합니다.
			- 특정 문제에 대해 자신만의 방식으로 플로우를 설계하고 구현한 좋은 경험입니다.
	

- 브라우저 및 OS 환경에 따라 스크롤바 노출 방식 차이로 발생하는 레이아웃 시프트 문제를 분석
	- overflow 조정, 여백 보정 등 기존 방식은 모두 스크롤이 레이아웃을 밀어내는 구조라는 한계 확인
	- 스크롤 영역이 레이아웃에 영향을 주지 않도록 OverlayScrollbars를 도입해 오버레이 방식 스크롤 구조로 개선
	- 그 결과 스크롤 유무와 관계없이 일관된 UI 레이아웃을 유지하도록 해결

- 인증 보안 강화를 위한 구조 개선
	- XSS 위험을 고려해 토큰 저장 방식을 localStorage에서 Cookie 기반으로 전환하는 의사결정을 제안
		- 인증 정보를 쿠키 기반으로 관리하여 HttpOnly,SameSite 처리하여 XSS 및 CSRF와 같은 보안 리스크를 낮춤
		- '보안' 까지 고려한 설계도 추가하면 조금도 완성도 높은 경험으로 보일 수 있습니다. (대부분의 신입 개발자 이력서에는 happy path 만 고려하는 경우가 많아서 이런점도 어느정도 차별성을 줄 수 있다 생각합니다.)
	- 모든 API 호출을 BFF를 단일 진입점으로 통합하고, Next.js를 리버스 프록시로 활용하여 토큰이 클라이언트에 노출되지 않는 구조로 설계
		- 여기 문장 구성을 정리하면 좋을 것 같습니다.
		- Next.js를 BFF로 구성 -> 모든 API 호출을 프록시 레이어로 설계 -> 성과
		- 성과는 아래 문장으로 충분히 표현되는 것 같습니다.

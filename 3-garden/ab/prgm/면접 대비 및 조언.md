## 🔍 비교 1: ESM은 "정적 import"

ts

복사편집

`// esm.js import { add } from './math.js'; console.log(add(1, 2));`

- ✅ `math.js`에서 `add`만 가져오는 걸 **코드를 읽는 순간 알 수 있음**
    
- ✅ 정적 분석 도구(트리쉐이커)가 "add는 쓰이고, subtract는 안 쓰이네" 판단 가능
    
- 💡 그래서 **사용 안 한 함수는 번들에서 제거할 수 있음**

## 🔍 비교 2: CommonJS는 "동적 require"

js

복사편집

`// commonjs.js 
const math = require('./math.js'); 
const op = process.env.METHOD; console.log(math[op](1, 2));`

- ❌ `math.js`에서 어떤 걸 쓸지 **실행 전에 알 수 없음**
    
    - `math['add']`일 수도 있고 `math['multiply']`일 수도 있음
        
- ❌ 정적 분석 도구는 "이 중 뭘 안 쓰는지" 판단 못 함 → **제거 불가**

## 🔁 왜 그런 차이가 있을까?

|요소|ESM|CommonJS|
|---|---|---|
|문법|`import { x } from 'mod'`|`const x = require('mod')`|
|호출 위치 제한|**최상단에만 import 가능**|어디서든 `require()` 호출 가능|
|분석 가능 시점|**빌드 시점에 모듈 관계 확정됨**|**런타임에 모듈 동작 결정됨**|
|트리쉐이킹 가능 여부|✅ 정적 구조라 가능|❌ 동적 구조라 불가능|

---

## ✨ 쉽게 요약하자면

- ESM은 **"이 JS 파일이 어떤 모듈에서 뭘 쓰는지"가 코드만 보면 딱 보임**  
    → 정적 분석 → 트리쉐이킹 가능
    
- CommonJS는 **"require를 함수처럼 사용해서 언제든 어떤 모듈이든 동적으로 가져올 수 있음"**  
    → 빌드 시점엔 파악이 안 됨 → 트리쉐이킹 불가

## 💬 정리 멘트

> "정적 구조라는 건 '코드 실행 전에 구조를 확정할 수 있다'는 뜻이고, 이 덕분에 번들러가 미리 어떤 코드가 쓰이는지 분석할 수 있어요. 반대로 CommonJS는 require가 함수처럼 동작해서 어떤 모듈을 언제 쓸지 미리 알 수 없기 때문에, 불필요한 코드를 제거할 수 없어요. 이 차이 때문에 Tree Shaking은 ESM에서만 제대로 작동하는 거죠."

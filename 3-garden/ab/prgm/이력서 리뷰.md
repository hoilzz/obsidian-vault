#### **프로젝트 설명 방식**
- **잘못된 접근:**
    - 단순히 “나 이거 개발했음”이라는 식의 나열.
        - 이 방식은 지원자의 역량을 보여줄 수 있는 정보량이 적습니다.
- **더 나은 접근:**
    - **문제 해결 과정과 사고를 어필하라:**
        - **어떤 문제를 만났는지?**
        - **문제의 원인을 어떻게 찾아냈는지?**
        - **어떤 방식으로 해결했는지?**
        - **해결되지 않았다면 그 이유는 무엇인지?**

- 유형 별 예시
	- anti: "무엇을 했다" 수준
	- 예시
		- Next.js을 이용한 서버 스트리밍 및 SSR을 적용 
		- 이것을 왜 도입했고 도입했을 때 어떤 효과가 있었는지 깊이있는 고민이 없음.
	- soso: 무엇을 했다 + 간단한 목적(혹은 흔한 이유)
	- 예시
		- Next.js SSR을 적용하여 SEO 최적화
		- 기술 적용 목적은 드러났지만, 구체적인 문제 맥락이나 혹은 왜 이것을 선택했는지, 이것을 통해 어떤 것이 개선 됐는지 (lighthouse를 통한 구체적인 수치도 좋다.)조금 더 나았을 것 같다. 혹은 이것과 더불어 nextjs 를 도입하면서 프로젝트에 미친 영향 혹은 개선된 것들을 추가로 적으면 좋을 것 같다. 
		- 이 이유 하나만으로 nextjs 를 적용할 수 있다고 생각한다. (더 나아가서는 그냥 next.js 다들 쓰니까) 라는 이유로 시작할 수 있지만 이 기술 스택을 사용하면서 어떤점을 개선했는지를 추가로 작성하면 차별성이 드러날 것 같다.
	- best-pattern: 문제 상황, 해결책, 이 해결책으로 인한 성과
		- 예시
			- SEO 개선:
				- 기존 CSR 구조의 SEO 개선을 위해 Next.js SSR로 전환하여 Lighthouse SEO 스코어 30 -> 95점으로 개선
				- long: 초기 CSR 기반 구조에서 크롤링 불가 문제가 발생하여, 특정 페이지의 콘텐츠가 구글 검색에 노출되지 않음. 이를 해결하기 위해 Next.js SSR로 전환하였고, Lighthouse SEO 스코어 30점 → 95점 개선. 제품 페이지 유입률 40% 증가.
			- 아래부터는 다른 케이스. 예컨대 nextjs를 단순히 요새 유행하니까 혹은 SSR로 SEO를 해결하고자 사용했지만.. 사용하다 보니 __next js의 서버 자원을 활용하여 다른 문제를 해결한 사례__ 즉, 이 기술 도입을 통해 특정 문제를 프론트 자원이 아닌 서버 자원을 활용하여 바운더리를 넓혀서 해결해본 경험이 있고 그 경험으로 프로젝트에 좋은 영향을 미친 사례. 
			- 인증 시스템
				- Next.js 서버를 리버스 프록시로 활용하여 JWT 인증 구조를 설계, SSR 환경에서도 안전하고 일관된 인증 흐름을 구성
				- long: 기존에는 백엔드에서 인증 쿠키를 관리했지만, 로그인 상태 유지와 SSR 동시 대응이 어려움. 프론트 주도 인증 설계를 위해 Next 서버를 리버스 프록시로 이용하여 jwt를 이용한 인증 쿠키 프로세스 구현. 이를 기반으로 Next 서버에서 안전하게 인증 토큰을 관리.
			- CI/CD: 
				- GitHub Actions 기반 CI/CD 파이프라인을 구축하고 Lighthouse CI를 연동해, 빌드 시점의 성능 지표를 자동 측정·관리할 수 있도록 개선

#### **2. 캐릭터라이징 (차별화)**
- **나만의 특색(방법)을 강조**
    - 단순한 작업 기록이 아닌, 자신의 **문제 해결 방식**과 **사고 과정**을 통해 본인의 장점을 드러낼 것.
    - 예:
        - "프로젝트 X에서 A라는 문제를 겪었고, 이를 해결하기 위해 B 방법을 선택했으며, C라는 성과를 얻었다."

#### **3. 강점에 집중**
- **이력서의 방향성:**
    - 모든 경험을 나열하려고 하지 말고, 자신만의 **강점과 자신감이 있는 부분**에 초점을 맞추자.
    - “나는 이런 문제를 해결할 수 있는 사람이다”라는 메시지가 전달되도록 구성.

### 기술 이력 어떻게 쓸까?
- **세부 사례와 깊이 있는 내용**을 담는 공간.
- 프로젝트에 대해 다음과 같은 내용을 포함:
    - 고민했던 부분: 어떤 문제가 있었고, 이를 해결하기 위해 어떤 노력을 했는지
    - 프로젝트를 하면서 느꼈던 점과 배운 점.
    - 팀 내에서 발생한 문제와 이를 해결했던 방식.
- 포트폴리오 혹은 블로그에 부연 설명을 위해 **코드 샘플이나 시각 자료**를 이용하여 구체적이고 명확하게 전달.

#### **2. 깊이 있는 고민의 중요성**

##### **Q. 고민의 깊이는 어떻게 보여줄까?**
- **자문하기:**
    - "왜 이 선택을 했는가?"
    - "다른 선택지가 있었다면 무엇이었고, 왜 그것이 아닌가?"
    - "결과적으로 이 결정이 프로젝트에 어떤 영향을 미쳤는가?"
- **사례 기반 서술:**
    - 단순히 "했다"가 아니라, **"고민의 과정"과 "결정의 이유"**를 포함.

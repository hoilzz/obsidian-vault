## 요약

1. 자기소개는 빠르게 확인
2. 경력 사항
   - 뭔가를 해봤다 -> 의미 없음
      - 해봤다면 WHY or 어떤 문제 떄문인지? 포함되면 좋다
         - __예컨대 UI/UX 개선: WHY가 중요__
         - msw 도입해봤다: 왜? 백엔드와 협업하는데 API 연동까지 일정이 항상 부족해서 미리 연동. 즉, 협업간 문제를 기술로 해결(아주 좋은 케이스)
3. 포트폴리오, 깃헙, 블로그 확인
   - 이 중 하나라도 없으면 평가 요소 부족으로 탈락 처리된다.
   - 깃허브
      - 프로젝트 코드 구조 파악. 문제 해결 과정이 담긴 PR 혹은 커밋 히스토리 파악
      - 코드 퀄리티는 모듈화, 유지보수 관점에서 변경에 용이한 추상화가 잘 되어있는지 
      - 어필하고 싶은 레포는 pin
   - 블로그
      - 단순 정보 나열 보다는 __자신의 경험과 생각을 담은글__
      - 개인적인 성찰, 경험이 부족한 경우 의미 있는 평가가 어렵다고 판단
         - 작성 내용에 대해 인터뷰시 답변 못하는 경우도 많다
      - 좋은글
         -  자신의 경험과 문제 해결 과정을 담은 글.
         - 특정 기술을 사용한 이유, 그로 인해 발생한 문제와 해결 방법, 배운 점 등이 포함된 글.
   - 서류 통과 및 면접시 얘기할 거리를 제공
그래서 서류단계에서 면접관이 검증하고 싶은 역량은? 
1. 같이 일하고 싶은 동료인가?
2. 같이 일하고 싶은 동료는 누구인가? -> 유지보수하기 좋은 코드를 작성. 단순 구현은 기본. 문제 해결 능력. 문제를 찾아내는 능력.(제품 뿐만 아니라 협업 과정에서도)

서류단계에서 중요한 포인트는 __차별화된 포인트와 나만의 색깔__ 이 있어야한다. 예컨대 이 친구는 FE지만 CI/CD 혹은 devops 에도 관심이 많은 친구. 디자인시스템 구축 경험이 많은 친구. e2e, 통합테스트 경험이 많은 친구. 협업 관점에서 문제를 잘 발견하는 친구 등..

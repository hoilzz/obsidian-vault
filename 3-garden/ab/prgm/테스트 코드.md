### 테스트와 SRP

#### SRP를 지키지 못한 컴포넌트

```tsx
import axios from 'axios';
import React from 'react';

function UserDashboard({ userId }) {
  const [userData, setUserData] = React.useState(null);
  const [isLoading, setIsLoading] = React.useState(false);
  const [error, setError] = React.useState(null);

  React.useEffect(() => {
    async function fetchData() {
      setIsLoading(true);
      try {
        const response = await axios.get(`/api/users/${userId}`);
        setUserData(response.data);
      } catch (err) {
        setError('Failed to fetch user data');
      } finally {
        setIsLoading(false);
      }
    }
    fetchData();
  }, [userId]);

  if (isLoading) {
    return <div>Loading...</div>;
  }

  if (error) {
    return <div>{error}</div>;
  }

  if (!userData) {
    return <div>No user data available</div>;
  }

  return (
    <div>
      <h1>User Dashboard</h1>
      <p>Name: {userData.name}</p>
      <p>Email: {userData.email}</p>
    </div>
  );
}

export default UserDashboard;

```

여러가지 책임을 가진 컴포넌트다.
1. 데이터 페칭
2. UI 렌더링
3. 상태 관리

1,3번은 react-query라고 봐도 된다. 이 컴포넌트를 테스트 하려면 API 성공/실패 시나리오를 모두 다뤄야 하므로 모킹 할게 많다. UI렌더링 테스트, 데이터 페칭 테스트가 엮여 있어 테스트 범위가 모호하다. 
ㅐ

중요한 결함을 사전에 방지할 수 있다.
		- 대규모 리팩터링, 아키텍처 개선, 주요 라이브러리 업데이트와 같은 **구조적 변경**이 이루어질 때, 사용자 시나리오 기반 테스트로 변경에 따른 결함을 사전에 차단할 수 있습니다.
		- **테스트가 통과**한다는 것은 주요 사용자 플로우에서 **사이드 이펙트가 없음을 검증**받았다는 강력한 신뢰를 제공합니다.  이는 안심하고 변경 사항을 배포할 수 있는 근거가 됩니다.

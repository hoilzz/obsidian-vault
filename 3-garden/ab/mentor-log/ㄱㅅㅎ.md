- 자기소개

		- 커뮤니케이션 역량
		- FE리딩
		- 사내 연구모임
	- 키워드(What)만 작성되어있어서 조금 아쉽습니다.
	- 꼼꼼함 -> __어떻게__ 꼼꼼하게 작업을 진행하는지 그리고 이로 인해 __어떤 효과 혹은 성과__ 가 있는지
		- 테스트 코드 혹은 TDD를 통해 명확한 요구 사항을 정의하고, 리팩터링시 기존 동작을 보장하는 과정을 중요시한다.
		- + 안정적인 제품을 만드는 기술적 습관으로 자리잡았다..

			- 정적 리소스 캐싱
			- 사실 이부분 내용은 오래전부터 웹서비스에 기본적으로 구축해야되는 내용이라 크게 차별성이 없는 것 같네요ㅠ_ㅠ

			- commit 메서드는 처음 들어보네요. 사실 어떤 내용인지 정확히 이해가 안되서 멘토링 때 다시 얘기해볼께요
			- 그리고 글만 봤을 때는 코드 레벨의 내용은 거의 스킵하고 넘어가는 경우가 많아요.
			- 코드 레벨의 내용보다 어떤 문제를 해결하기 위해 어떤 업무를 했는지 혹은 그 작업으로 인해 팀에 어떤 성과가 있었는지가 더 중요합니다.

### 3-3. 이력서 총평

- 기능 위주의 소개
- 특징에 적어준 기술 내용은 깊이가 조금 부족하다. 
- 현재 이력서의 내용을 수정한다기보다 조금 더 깊은 고민이 드러날 수 있는 성과가 필요하다.
	- 제품을 운영한다는 관점에서 개선할 수 있는 것들을 찾자.
		- 공통 컴포넌트 관리 (디자인 시스템 혹은 헤드리스 컴포넌트)
			- 이 컴포넌트를 관리하기 위한 기법들
				- SOLID
				- compound component pattern
			- 이 컴포넌트의 유지보수를 위한 테스트 코드
				- 리그레션 테스트를 줄일 수 있는 의미있는 테스트
				- 리팩토링을 안전하게 할 수 있는 장치 역할을 할 수 있는 테스트
		- 협업간 발생하는 병목이나 중복들이 뭐가 있을까?
			- DTO 작성 자동화
	- 그리고 최신 트렌드는 AI가 묻어야한다.

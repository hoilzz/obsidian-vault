### 6 질의응답
1. 제  이력서의 어떤 부분을 보완하면 가장 효과가 큰지 피드백 받고 싶습니다.  <br>요즘 신입 지원자들의 수준이 전반적으로 높아진 것 같아서, 제 이력서가 경쟁력 있게 보이려면 어떤 포인트를 더 보완해야 하는지 궁금합니다. 
	- 단순 트러블 슈팅 내용보다 설계/시스템에 관한 문장이 많아야한다.
	- 지금 내용은 사건 서사(문제→해결)가 길어. 이력서는 “일기”가 아니라 “의사결정과 시스템”을 보여줘야 함.
	- hydration mismatch: 길게 쓰지 말고 1줄로 “SSR/CSR 상태 일관성 보장 패턴 적용”
	- 워터폴: “dehydrate/hydrate 캐시 재사용으로 데이터 플로우 재구성”
	- WebView: “브릿지 프로토콜 설계 + 인증 race condition(single-flight/queue) 해결”
	- 디자인시스템(헤드리스), 테스트 자동화, 
- 강점: 자동화 
	- PR required checks (lint/typecheck/test/build/Chromatic)
	- 캐시 최적화로 CI 단축
	- 도커 이미지 빌드 -> 컨테이너 띄우고 -> 쿠버네티스로 오케스트레이션 gitops
	- Playwright e2e or backstop 2~3개만 추가(로그인/리스트/상세)
		- “UI 불일치/흐름 단절을 시스템으로 막는다”가 말이 아니라 **파이프라인**이 됨.
		- 즉, **‘버그를 고쳤다’가 아니라 ‘구조를 바꿨다’**로 써야 고평가됨.
	- AI 시대에 맞춘 “한 단계 레벨업” 방향 (FE 관점에서 현실적인 것)
		- “모델 호출” 자체가 아니라 **AI 기능을 제품으로 만들고 운영**
		- “AI로 개발 생산성”을 시스템화: PRD→UI scaffolding 파이프라인
			- 너가 이미 CI/표준화 쪽이 강하니까, AI를 “개발 워크플로우”에 붙이면 이력서가 확 튐.
			- PRD(또는 요구사항) 입력 → 컴포넌트/스토리북 스토리/테스트 스캐폴딩 생성
			- 생성물은 **Storybook+Chromatic**으로 검증되게 연결
	- RAG/검색 품질 + 평가(Eval)까지 한 번이라도 해보기
		- 요즘 “챗봇 만들어봤어요”는 너무 흔함. **평가/품질 관리**를 넣으면 차별화됨.
		- 간단한 문서(RFC/FAQ) 기반 RAG
		- 답변 품질을 체크하는 **작은 eval 세트**(10~30개 Q/A) 만들고 회귀 테스트
		- “정확도/근거 링크/응답 시간” 같은 지표를 대시보드로
		- n8n으로 하면 너무 쉬움
    
1.  지원할 때 좋은 공고를 고르는 기준을 알고 싶습니다. 
	- ‘일의 범위와 기대치가 구체적인 공고’
		- 좋은 신호는 채용 이유가 명확합니다(어떤 기능/영역을 확장하기 위해 FE가 필요). 스택이 왜 필요한지 설명이 있습니다(Next.js를 쓰는 이유, SSR/SEO가 필요한지, 모노레포/테스트/CI가 있는지). 협업 구조가 드러납니다(PM/디자이너/백엔드와 인터페이스가 어떻게 구성되는지). 품질 기준이 있습니다(배포 프로세스, 모니터링, 장애 대응 방식).
	- 신입이면 “성장”보다 ‘리뷰/배울 구조가 있는 팀’이 더 중요
	- 좋은 팀은 코드리뷰 문화 + 배포/모니터링 체계가 최소한 있음.
	- 이에 대한 힌트는 회사 혹은 팀 블로그를 통해 알 수 있다.
	- 이게 없다면 링크드인을 통해 해당 회사에 재직중인 사람을 찾아 그 사람의 개인 블로그 혹은 활동을 통해 유추해야한다.
	- 그래도 없다면 알 수 있는 방법과 근거를 찾을 수 없다. 면접을 통해 유추해내는 수밖에 없다.(지원자에게 질문하는 시간을 이용.)
2. 최근 채용 공고를보면 AI Agent를 사용한 경험을 묻는 공고가 많이 보이는 거 같습니다.프론트엔드 입장에서 AI Agent를 어떤 식으로 활용하면 강점으로 보일지 궁금합니다.
	- 채용에서 강점이 되려면 “AI를 써봤다”가 아니라 “개발 프로세스를 자동화/표준화해서 품질과 속도를 올렸다”가 되어야 합니다. 즉, 도구 사용 경험이 아니라 시스템을 만든 경험으로 보여야 합니다.
	- 강하게 보이는 방향은 세 가지가 많습니다.
		- 첫째, 품질 자동화입니다. PR에서 변경된 컴포넌트만 대상으로 시각적 회귀 테스트를 돌리고 결과를 자동으로 요약하거나, 테스트 실패 원인을 정리해 리뷰어에게 제공하는 방식입니다. 단순히 테스트를 돌리는 게 아니라 팀의 리뷰 비용을 줄이는 방향이 강점입니다.
		- 둘째, 문서화/온보딩 자동화입니다. 컴포넌트의 props, 사용 예시, edge case를 코드에서 추출해 문서나 스토리북 설명을 자동 생성하고, 디자인 토큰이나 API 스키마 변경 시 영향 범위를 리포트로 뽑아주는 식입니다. 이것도 팀 전체 생산성에 직접적인 영향을 줍니다.
		- 셋째, 컨벤션 준수 자동화입니다. 린트/타입 에러를 고치는 수준이 아니라, 프로젝트 규칙에 맞는 패턴으로 리팩터링을 제안하고 PR을 생성하거나, 코드리뷰 체크리스트를 기반으로 변경사항을 검토해주는 식의 활용이 더 설득력 있습니다.
	- 최근 트렌드는 오케스트레이션
		- 스캔(PRD) -> tech-SPEC 작성 -> 작업 시작 -> 검증
		- 메인(오케스트레이터) 에이전트가 목표를 작업 단위로 분해하고, 각 작업을 담당하는 서브에이전트/스킬/툴을 호출해 순서·의존성·검증·재시도까지 관리하면서 결과물을 합성하는 운영 방식.
		- 예시 1: “UI 구현 워크플로우 오케스트레이션”
			- 메인 에이전트: ‘Figma 링크 + 요구사항’을 받아 구현 플로우를 제어
			- 서브/스킬
			    - 디자인 토큰/컴포넌트 매핑(DS 규칙 적용)
			    - 레이아웃/컴포넌트 코드 생성
			    - 상태/데이터 로딩 방식 제안(SSR/CSR, 캐시 전략)
			    - 스토리북 스토리 생성
			    - 테스트 케이스 초안 생성
		- 결과: 단순 생성이 아니라 “조직 규칙을 반영한 구현”으로 연결
		- 예시 3: “장애 대응/원인 분석 오케스트레이션”
			- 메인 에이전트: Sentry 이슈나 로그를 받아 트리아지
			- 서브/스킬
			    - 에러 재현 조건 추출
			    - 영향 범위(브라우저/버전/라우트) 분류
			    - 의심 커밋/코드 영역 추적
			    - 핫픽스/롤백 제안
			    - 회귀 방지 테스트/모니터링 항목 생성
			    - 결과: 대응 속도와 품질을 동시에 올림
	- ‘서브 에이전트가 여러 번 불린다’가 아니라, 제어가 있어야 한다
		- 언제 분해하는지
		- 어떤 기준으로 병렬/직렬 실행하는지
		- 실패하면 어떻게 재시도/폴백하는지
		- 결과를 어떤 포맷으로 합치는지
3. 작더라도 실사용성을 가진 개인 프로젝트를 만들고 싶습니다. 실제로 사람들이 이용하게 만드려면 어떤 요소들을 챙겨야 하는지 궁금합니다.  
	- 실사용 서비스는 기능보다 운영 요소가 더 중요합니다. 작은 서비스라도 다음을 최소로 갖추면 ‘실사용’ 가능성이 올라갑니다.
	- 민수님 본인이 사용할 서비스를 만들면된다. 대단한 서비스가 아닌 내가 겪는 문제를 해결.(작게작게. 일정 조율, 스터디 출석/벌금, 개인 운동 기록)
		- 혹은 미국, 일본 등 주요 선진국의 Top10 앱을 다운받아서 그 앱보다 조금 더 낫게 혹은 벤치마킹하여 한국에 배포
		- 실사용성이라 함은 사람들의 문제나 병목을 더 간편하게 해결해주는 것들이라고 생각.
		- 저도 그래서 바이브코딩으로 가계부앱과 주식 일기 앱 만드는중.. 
		- 자기가 좋아하는 분야에서부터 시작하면 될듯.
	- 로그인과 계정 관리(최소 소셜 로그인, 계정 삭제). 
	- 에러 수집과 모니터링(런타임 에러를 추적할 수 있어야 개선이 가능합니다). 
	- 성능과 기본 SEO(웹 서비스면 OG/sitemap/핵심 지표 점검). 
	-  피드백 루프(사용자가 불편을 남길 창구). 
	-  최소한의 지표 수집(어디서 이탈하는지 알아야 개선이 됩니다).
4. 스타트업 신입 채용에서는 학벌을 어느 정도로 중요하게 보는지 그리고 학벌보다 더 크게 평가되는 요소가 무엇인지 궁금합니다.  
	- 스타트업에서도 학벌이 전혀 영향이 없다고 말할 수는 없습니다. 지원자가 많을 때 초기 필터로 작동할 수 있습니다. 다만 대기업보다 비중이 낮고, 실력이 증명되면 학벌은 빠르게 무력화됩니다.
	- 학벌보다 더 크게 보는 것은 다음입니다. 첫째, 구현 능력의 증거(배포 링크, 운영 경험, 사용자 피드백 반영). 둘째, 문제 해결 서사(왜 발생했고 어떻게 찾았고 왜 그 방식으로 고쳤는지). 셋째, 협업 신호(PR, 리뷰 경험, 문서화, 커뮤니케이션 방식). 넷째, 성장 속도(짧은 시간에 복잡한 문제를 해결한 흔적).
	- 이 이력서는 성능/구조적 문제 해결 경험이 있어서 학벌이 약해도 충분히 경쟁력을 만들 수 있습니다. 다만 앞에서 말한 “서술 구조와 주체성”이 강화되어야 그 강점이 제대로 보입니다.
5. 취업 후 실무 경험을 쌓으면서도 꾸준히 공부를 이어가고, 나중에는 카카오, 토스, 배민 같은 대형 서비스 기업으로 이직하는 걸 목표로 하고 있습니다.  <br>그 목표를 이루려면 어느 정도 수준까지 준비되어야 하는지, 그리고 어떤 방향으로 성장해나가는 게 좋은지 조언을 받고 싶습니다.
	- 대형 서비스 기업은 “몇 년 했는지”보다 “어떤 임팩트를 어떤 기준으로 냈는지”를 봅니다. 프론트엔드 기준으로 핵심은 네 가지입니다. 
		- 웹 관련 성능 최적화 (렌더링/네트워크/캐시/성능/접근성), 
		- 데이터/상태 관리(서버 상태와 캐시 전략, 비동기 경합 처리), 
		- 품질 시스템(테스트, CI, 모니터링, 회귀 방지)
		- 설계/추상화(재사용 가능한 컴포넌트/패턴 설계, 확장성, 팀 생산성 기여).
	- 현실적인 성장 방향은 이런 흐름이 좋습니다. 
		- 첫 단계는 제품 운영을 해보는 것(배포, 장애, 로그/모니터링, 사용자 이탈 지점 개선). 
		- 다음 단계는 팀 생산성을 올리는 경험을 만드는 것(공통화, 표준화, 문서화, 자동화). 
		- 그 다음은 기술 의사결정을 주도하는 경험(트레이드오프를 설명하고 결과로 증명).
	- 이직을 목표로 한다면, 개인 공부는 넓게 펼치기보다 “내가 만든 개선이 팀/서비스에 남았다”는 증거를 만드는 쪽이 효율적입니다. 예를 들어 성능 개선, 품질 자동화, 공통 컴포넌트/패턴 도입처럼 재사용성과 지속성을 가진 작업이 가장 강한 포트폴리오가 됩니다.

					- Old: 오래 살아남은 객체가 이동됨. 마크엔스윕(루트 객체에서 접근 가능한 객체 마크, 마킹 안된얘는 스윕(삭제))

- Graceful shutdown
	- __`socket.destroy()`로 강제 종료했을 때 발생할 수 있는 문제점?__
		- 진행 중인 요청 손실, 네트워크 에러 증가(LB가 해당 노드를 비정상으로 인식할 수 있음)
		- 일정 시간 후 강제 종료. 
	- __server.close 후 process.exit() 바로 호출?__
		- Node 이벤트 루프가 비워질 때까지 기다리자
		- setTimeout(0) 을 활용해서 이벤트 루프 비워질 때까지 시간을 주자.
			- Kubernetes가 **Pod을 종료할 때** 먼저 `SIGTERM`을 프로세스에 보냄.
		    - Next.js 서버가 `SIGTERM`을 특별히 처리하지 않더라도, Node.js 프로세스가 이를 감지하고 **새로운 요청을 받지 않음**.
		    - 하지만 진행 중이던 요청은 종료되지 않기 때문에, **완전히 안전한 종료는 아님**.
		    - **Load Balancer(LB)와 K8s의 `preStop` Hook이 일부 보호**
			    - K8s의 **Service 및 Load Balancer가 종료 중인 Pod으로 트래픽을 보내지 않도록 함.**
			    - 다만, `server.close()`를 호출하지 않으면, 기존 요청이 처리되는 동안 강제 종료될 수도 있음.
			    -     - K8s는 기본적으로 **30초 동안 기다린 후 SIGKILL로 강제 종료**함.
			    -     - 이 시간 안에 **Node.js가 자연스럽게 종료되면 문제없음**, 하지만 긴 요청이 있으면 중단될 수 있음.
	

    

```ts
server.close(() => { 
	console.log('Server closed.'); 
	setTimeout(() => { 
		console.log('Process exiting...'); 
		process.exit(0); 
	});
});

```

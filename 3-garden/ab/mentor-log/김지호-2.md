### 추가 피드백

- 일단 모든것에 이유가 있어서 좋다
- 코드에 일관성이 있어서 몇가지 파일을 보면 전체 컨셉이 이해가 되서 이해하기 쉬워진다.
	- 폴더구조 좋음 네이밍이 쪼금 아쉽다. 
	- 전체 구조는 확실히 이해가 잘됨. 디테일이 중요. 
		- 일단 next의 page.tsx -> Client.tsx -> useActions -> Handler
		- 이렇게 일관성 있는 패턴은 좋음. 근데 actions와 handler가 와닿는 네이밍은 아닌 것 같아요. 
		- actions가 어떤 파일은 API호출을 하고, 어떤 파일은 단순 hook이어서..
		- 차라리 hooks로 하는게 덜 헷갈릴 것 같습니다. actions 네이밍 자체가 넥스트와 관련된 서버 액션인가? 그럼 API를 호출하는 함수인가? 라는 헷갈릴 수 있는 여지를 주는 것 같습니다! handler도 비슷한 이유로 조금 헷갈립니다.
		- 차라리 API호출은 확실히 useXXXquery, useXXXmuation
		- API와 연관 없는 hook은 그냥 hooks/아래에 useXXX로 네이밍 지어도 될 것 같습니다!
- handler 네이밍이 apis에서도 쓰이고 useAction에서도 쓰여서 헷갈림
- common/token 이 api 뭐길래 계속 쏘지?
- PC버전이 앱라우터 효과보기 좋다.
	- 앱라우터 왜쓴다 생각?그리고 loading.tsx 가 없는데 이유가?
		- loading.tsx 까지 추가하여 next가 가이드하는대로 스트리밍 적용 가능.
- 이미지 레이지로드.. 게시물을 엄청 많이 만들어보자 + 무한스크롤도 구현해보자.
- 컴포넌트의 데이터와 UI의 분리
	- UI는 그냥 prop받는대로 그리자
		- 이렇게 하면 재사용성 + 테스트하기 쉬운 코드가 됨.
		- 그리고 UI가 예측이 엄청 쉬워짐. 그냥 얘는 받는대로 그리니까
	- 데이터는 page나 혹은 page 아래에 레이어(container) 라는 레이어를 두고 여기서 로직 작성하여 component에 다 내려주자.
- Header에서 pathname에 따라 UserProvider 렌더링 여부를 결정하고 있습니다.
	- 요건 nextjs의 group-route를 활용하여 로그인 유저일 때 레이아웃, 미로그인 유저일 떄 레이아웃을 구분하여 사용하면 쉽게 해결 될 것 같습니다.
	- 현재는 path로 결정하고 있어서.. 사이드이펙트 나기 쉬운 코드로 보입니다.
- 추가로 Userprovider 보면 user에 대한 데이터를 서버쪽에서 prefetch하고 있지 않습니다.
- 유저 정보(토큰)와 같이 보안이 중요한 건 서버쪽에서 호출을 하여 하이드레이트로 내려주는 걸 권장드립니다.
	- 그 때 코드 보면서 추가로 피드백 드렸던건.. next의 fetch 함수로 중복하여 호출하더라도 기존에 호출된 데이터가 있으면 캐싱이 되어있기 때문에 여러번 호출하더라도 1번 호출되도록 최적화가 되어있습니다. 참고 부탁드려요!

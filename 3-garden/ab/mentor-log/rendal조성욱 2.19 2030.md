- react -> nextjs 마이그레이션
	- app router를 썼는지
		- page router와 app router의 차이점
	- 어떤 성능개선이 있었는지
		- 라이트하우스로 LCP, FCP 측정해봤는지
	- 중구난방 컴포넌트 구조, 중첩 HOC -> hook, DDD구조 도입
		- 간단하게 설명
	- 어느 부분을 어떻게 개선하면 좋을지 고민
		- 이것에 대해 설명
	- 서비스 성능 혹은 팀의 작업 효율성을 높이는데 기여했는데 설명
- 오니온 파이브 프론트 전반적인 시스템, 코어,템플릿 설계하여 재사용성 효율성 극대화
	- 전반적인 시스템을 어떻게 개선했는지
	- 코어, 템플릿 구체적 설명
	- 폴더구조가 어떻게 개선된건지?
- 최신기술을 사용해 서비스 최적화, 구조적 개선
	- 최신 기술을 사용하면 무조건 서비스 최적화가 되는것일까? 
		- 기대하는 답변은 아님. 단순한 정적 페이지면 react로만 작업해도 괜찮지 않을까?
	- 최신 기술이 좋다는 식의 답변은 좋지 않음.
		- 비즈니스, 기술, 팀 생산성 관점에서 균형 잡힌 답변이 필수다. 여기서 가장 중요한건 비즈니스
			- 베스트 답변은 비즈니스적 가치를 우선으로 하고 그 다음 장기적인 유지보수와 팀 생산성을 고려한다.
		- 예컨대, SSR을 활용한 SEO 최적화 혹은 SSR, ISR과 같은 성능 개선과 같은 명확한 이점 
			- ISR, SSR언제 어떻게 사용하면 좋을까요? 
		- 팀 생산성 관점에서는 기존 코드와의 호환성, 장기적 유지보수 부담도 고려
- 기술의 원리
	- React의 hydrate
		- hydrate에 대해 설명해주세요
		- SSR: 서버에서 리액트 컴포넌트 미리 렌더링해서 완성된 HTML을 클라에 보냄
			- 이때 서버는 React를 실행하여 HTML을 생성
		- 브라우저는 이 HTML을 보여주면서 React는 hydrate를 통해 기존 HTML을 리액트 컴포넌트로 변환하여 이벤트 리스너 연결
			- 연결 전에는 버튼 클릭같은 인터랙션 동작안함(리액트 코드 미실행됨.)
		- 그래서 클라에서 리액트 번들 다운로드 및 실행
			- 리액트 애플리케이션의 JS 번들 로드됨
			- 이 번들에 리액트 컴포넌트, 이벤트 핸들러가 포함됨
		- React의 hydrate 실행 -> 기존 HTML에 리액트 이벤트 연결
			- hydrate는 기존 HTML을 파싱, 리액트의 가상 DOM을 ㅣㅂ교하면서 필요한 이벤트 핸들러만 추가
		- 초기 로딩이 빠르지만 상호 작용 가능하려면 hydrate필요
		- SSR에서 hydrate 핵심역할
			- 기존 HTML 유지하면서 리액트 컴포넌트와 연결
			- 리렌더링 없이 이벤트 리스너만 붙여서 상호작용 가능하게 만듬
			- 초기 로딩속도 빠르게 유지
			- 그래서 page router는 모든 컴포넌트가 클라에서 실행되기 떄문에 hydration 필요. app router는 클라 컴포넌트가 없다면 hydration 없음. (정확히는 클라에서 실행할 JS코드가 없는 경우, 단순한 p태그 이런것만 그리면 여튼 하이드레이션 없다.) 그래서 JS번들도 없는 경우가 있음.
	- qna
		- app router 
			- 서버컴포넌트만으로 렌더링된 부분은 hydrate 노필요
			- 근데 use client 즉 클라 컴포넌트가 포함되면 하이드레이션 필요
- 리덕스와 주스탄드 차이
	- 걍 저는 보일러플레이트 정도 차이같다.

## 2. 지원자의 현재 상황과 목표 체크


  
- Q1. 이직을 대비한 **이력서 피드백**, **기술 면접 준비**와 **커리어 상담** 
- Q2. Web3 메타버스 게임사 캐리버스에서 1년 1개월 경력이 있습니다. 블록체인 개발 직무로 입사하였습니다. 
	- 블록체인 서비스 운영 지원을 주로 하였고 개발 부문에서는 신규 프로젝트로서 같이 진행하는 부분이 많았고 프론트엔드 부문을 담당하여 개발을 진행했습니다. 
	- Admin 사이트 리뉴얼, 텔레그램 웹 게임 개발에 참여한 이력이 있습니다. 
- Q3. 현재 중견기업 이상으로 이직하는 것을 목표로 하고 있습니다. 잘 부탁드립니다!


회사에서는 신규 프로젝트를 4-5가지 가량 진행해오면서 구현 위주로만 프론트엔드 작업을 진행해와서 성능적인 측면을 고려해서 진행해오진 못했습니다. 그래도 블록체인 내용을 제외하고 의미있다고 생각되는 부분을 가져와서 서술했어요.  
  
이력서, 경력기술서는 어떤 방식으로 서술하면 좋을지 몰라서 두 버전으로 나누어서 작성했습니다!  
  
포트폴리오는 이제 개발이 끝나는 상황이라 아직 문서화하진 못했어요.  포트폴리오에 대한 피드백도 간략히 해주시면 감사하겠습니다!  해당 이력들 보시고 도움될 만한 부분이나 면접 예상 질문도 생각나시는 부분이 있으면 말씀 부탁드립니다!

#### 경력기술서1

- 회사 규모, 이직 사유 제거해도됨.
- 사용기술은 각 프로젝트에 작성되어있으니 생략하면 좋을 것 같다.
	- 경력 기술서는 임팩트있는 1줄이 필요.
- HTML 게임 개발
	- 사실 눈에 띄는거 없음. 단순 도입. 
		- 경력에 도움되는 일의 접근방식 중 하나는 협업간의 문제 해결
			- 하지만 eslint, prettier, husky 그리고 fsd
			- 흔한 경험이자 이건 누구나 하는 작업.
	-  프로젝트 기간이 짧은거 같은데 얼마나 되나요?

- 사내 관리 도구 개선
	- 1번 항목이 괜찮아 보인다. 이건 사내 어드민을 개선한 좋은 사례로 보인다. 마찬가지로 접근은 좋지만 매력도는 떨어진다. 보는 사람에 따라 평가가 달라질 것 같다.
		- 매력도가 떨어지는 이유는 CSV 문서를 이용해 요청하는 프로세스로 보여서
			- 이상적인 상황은 지라 도입인데.. 이것 또한 단순 도입이라 이걸 어필하기 쉽지 않아 보인다.
		- 내가 이해한게 맞는지 모르겠음. 
			- 요청 프로세스가 딱히 없었는데 (그냥 대화로 넘긴 느낌?) CSV 파일로 문서화 한게 맞나?
	- 2번 항목은 확장 프로그램 사용중 -> 어떤 확장 프로그램인지
		- 응답 값에 대한 핸들링 로직 부족 -> 평가자 입장에서 이해하기 어렵다.
		- 대충 이해한건 동작하지 않는 버그가 있을 때 초기화 로직을 추가하여 새로고침 없이 동작 가능 인 것 같다
			- 버그가 발생 -> 초기화 (문제 해결 방식이 쪼금 좋은 평가를 받기 어렵다.) 버그 생겼으니까 그냥 초기화 할께요~ 로 읽힌다.
	- 3번은 단순 역할 부여인데 혹시 기술적으로 난이도가 있는 작업일까요? 난이도가 있다면 그걸 더 표현할 방법을 찾아볼께요
	- 4번 테스트 환경 서비스를 구축한 건 좋은 것 같다. 
> - **환경 변수 및 빌드 설정 분리 자동화**
    > `.env.production`, `.env.staging`을 분리하고 CI/CD 파이프라인에서 자동으로 주입되도록 구성하여 환경 혼선을 방지했습니다.
- **브랜치 기반 배포 자동화 (CI/CD 통합)**
    > Git 브랜치(`main`, `develop`, `feature/*`) 기준으로 Vercel Preview / AWS Amplify 환경을 자동 배포하여 QA 및 코드 리뷰 프로세스를 시각화했습니다.
- **테스트 환경용 모의 데이터(Mock API / Stub 서버) 구축**
    > 백엔드 연동 전, MSW(Mock Service Worker)를 활용해 독립적인 프론트엔드 테스트가 가능하도록 구성했습니다.

- Web3 교환소 고도화 및 실시간 운영 장애 대응 
	- 사실 1,2번은 단순히 참여 및 깨닫는 얘기인데.. 이정도 내용은 생략하는게 좋다. 좋은 평가를 받는데 도움이 되지 못한다.

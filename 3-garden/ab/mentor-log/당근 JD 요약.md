#### 이지픽 
- 지수 백오프 알고리즘 왜 도입?
	- 이거 그럼 사용자는 무슨 페이지를 보고 있는지?
	- 이걸로 옮긴지 꽤 됐는데도 아마존에서는 얘가 여전히 대규모 문제들을 여러번 일으켰어요. 그래서 다른 솔루션이 제시됐어요. Token bucket 을 쓰는건데, 아직도 exp backoff가 제일 흔한거같긴 해요.
	- 한계점은 뭐라 생각하는가?
	- 에러 원인이 서버가 이미 너무 많은 트래픽을 받는 경우라면 retry가 문제를 심화시키지 않을까?
	- 혹은 무한 재시도가 괜찮은 방법일까?
	- Adaptive throttling: 과부하 때 클라가 요청 성공률을 기반으로 스스로 요청을 제한(스로틀링)하는 기술
		- 서비스 과부하에 서버 부하 줄이는데 효과적
		- 요청거부확률 
			- 요청수와 성공한 요청수가 거의 같다면 0에 가까워져 클라는 계속 요청 보냄
			- 0보다 커지면 거부 확률이 증가하면서 이 확률에 따라 일부 요청 의도적으로 막음
		- 요약하자면 내가 요청 많이 보내는데 잘 안되는거 같아. 서버 부하만 늘어나니 잠시 쉬었다 하자.
		- 이 방식은 특히 **FE 클라이언트가 많은 요청을 보낼 가능성이 있는 환경**(예: 실시간 데이터 요청, 대규모 사용자 트래픽)에 적합합니다.  
		- 지수 백오프의 단점을 보완하면서, 클라이언트가 서버 부하에 **적응적으로 반응**한다는 점이 매우 효과적이라고 생각됩니다.
		- 다만 사용자 경험: 왜 대기해야하는지.. 
		- 지수백오프를 구현해보세요 할지도
```
const fetchWithExponentialBackoff = async (url, options, retries = 5, delay = 1000) => {
  try {
    const response = await fetch(url, options);
    if (!response.ok) throw new Error("Request failed");
    return response.json();
  } catch (error) {
    if (retries === 0) throw error; // 재시도 한계 도달 시 에러 던지기
    const backoff = delay * 2; // 지수 증가
    await new Promise((resolve) => setTimeout(resolve, backoff)); // 대기
    return fetchWithExponentialBackoff(url, options, retries - 1, backoff); // 재귀 호출
  }
};

```

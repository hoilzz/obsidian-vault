### 부족한 경험과 성과
- 정답은 없다. 문제 정의와 해결 과정을 자신만의 기준으로 해나가면서 경험을 쌓아나가는 것이 첫번째.
- 일단 경력 자체가 짧다. 1년 이상 서비스를 길게 운영하면서 겪는 이슈들이 공통적으로 있다
- 서비스를 운영하면서 코드의 유지보수 높이기 혹은 코드의 안정성
		- 서비스를 운영하면서 프로젝트가 커지는데 FE쪽에서 주로 겪는 컴포넌트의 재사용성을 높이는 작업
			- 예컨대 디자인 시스템 구축, 컴포넌트 추상화 하여 컴포넌트 설계와 관련된 내용이 보이질 않음.
		- 서비스를 운영하면서 nextjs의 캐싱 전략을 서비스 상황에 맞게 어떻게 녹여냈는지
		- 다양한 테스트를 통해 코드의 안정성을 높인 경험
			- 개발자 관점에서 대규모 리팩터링 및 구조개선, 주요라이브러리 업데이트시 주요한 시나리오 잘 동작하는지를 e2e로 미연에 방지 가능(CI/CD 때 테스트 자동화).  혹은 통합 테스트를 통해서도 미연에 방지가능
			- 새로운 기능 개발시에 테스트 코드를 통해 코드가 예상대로 동작한다는 것에 대한 보장을 제공
				- 이 기능 개발시에 요구사항 변경 혹은 코드 리팩터링이 발생할 수도 있는데 이에 대한 코드 안정성 제공
		- 은서님은 풀스택으로 서버 개발 경험도 많으니까 rest api 단점을 gql 도입으로 개선했다. 할 수도 있구요.
	- 자란다에서 정확히 어떤 걸 개발했는지 알 수 없음
		- 어떤 페이지를 개발했다 정도를 적어줘야 어떤 도메인 지식을 가지고 있을지 예상이 될 거 같다
		- 예컨대, 커머스로 치면 상품 리스트나 상세 페이지, 검색, 주문, 결제 등이 있지 않는가
			- 이렇게 됐을 떄 이 사람이 어떤 작업을 했고 어떤 도메인 지식이 있을지 예상이 됨.

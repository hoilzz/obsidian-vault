### 자란다

자란다 설명 부분 좋음. 서비스 설명, 현재 팀 여건이 어렵지만 극복하기 위한 노력들을 하고 있다로 잘 보입니다.

- 코드아키텍쳐 리팩토링
	- 설명도 잘 읽혀요 가독성 좋아요. 블로그 글 읽고 싶은 내용입니다.
	- React 아키텍처를 고민해보다 글 리뷰
		-  _솔직히 처음 입사했을 때 부터 마음에 안드는 코드와 정책이 많았는데요._
			- 요 멘트는 쪼금 공격적으로 보여요. __개발을 진행하면서 개선할 수 있는 포인트들이 많이 보였습니다.__ 와 같은 느낌이 좋을 것 같아요. 부정적인 것보다는 개선 가능성과 같은 긍정적인 느낌으로 작성해주세요! 이런 느낌의 문구는 은서님을 채용하기에 조금 꺼려할 수 있어요
		- _1. Service 역할 불필요_
			- 음 일단 잘 읽혀요. 어떤 문제가 있는지 알 수 는 있습니다. 음 괜찮으시다면 현재 코드를 TODO LIST와 도메인으로 대체하여 예제코드가 있으면 좋을 것 같기도 해요 ㅎㅎ (간단하게)
			- 이거 제목에서 Service 역할이 왜 불필요한지 혹은 어떤것이 정확히 문제인지 키워드 하나로 나타낼 수 있으면 좋을 것 같아요. (나머지 2번~ 도 다 마찬가지입니다.)
		- _2. Entity 혹은 Model의 부재_
			- 일단 이것도 잘 읽힙니다. 어떤 문제가 있는지 잘 확인이 돼요.
			- 이것도 동일하게 예제코드.. 투두리스트나 누구나 아는 도메인의 서비스를 예로 들어서 코드 간단한 예시 있으면 좋을 거 같아요.
		- _3. Page 컴포넌트의 비대함_
			- 팀의 정책 중 가장 납득하기 힘든 정책 -> 이것도 부드러운 워딩으로..ㅎㅎ
			- 테스트에 관한 얘기도 좋았어요.
			- 제목을 Page 컴포넌트가 너무 많은 책임을 가지고 있어요 (feat. SRP) 뭐 이런느낌..인데 전 옛날사람이라 이런게 괜찮은거 같기도하고..ㅎㅎ 제목만 보고도 두괄식 형태로 이해되면 좋을 것 같아 말씀드렸어요!
		- _4. 분리 자체가 명확하지 않음_
			- 요기는 쪼금 제목이 모호한거 같아요
				- "어떤 기능을 수정/추가하려면 어느 페이지 폴더를 가야 하는지" 찾기부터가 복잡해질 수 있습니다.
				- 이 문장으로 4번은 요약할 수 있을 것 같아요. 팀의 DX를 떨어뜨리는 컨벤션.
				- 차라리 제목에 약간 가벼운 느낌으로 "어떤 기능을 변경하려면 어느 페이지로 가야하는지 모르겠어요" 뭐 이런 느낌도 괜찮을거 같아요(이건 은서님 취향대로..ㅎㅎ)
		- _고민 내용 1. React에 의존적으로 짜지 않아야 하나?_
			- 문제 정의와 선택지 중 하나의 단점을 찾고 다른 선택지를 선택하고 그 효과와 이유 설명한점이 좋았어요
		- _고민 내용 2. 기능 및 컴포넌트를 Domain별로 묶을까?_
			- 이것도 내용이 좋았습니다. 두가지 선택지중 장단점을 비교하여 선택하는 과정이 좋아요.
		- _현재 개발중인 아키텍처_
			- 글이 잘 읽히고 팀에 어떤 문제를 찾아서 개선하는 태도도 너무 좋습니다!! 제 피드백을 잘 생각하고 고민하신 결과가 보여요 ㅎㅎ 
			- 요거에 대해 개선을 어떻게 했는지 후기도 남겨주시면 베스트일 거 같아요.

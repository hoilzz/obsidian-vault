### 3-2. 이력서 리뷰
- _자기소개_
	- 자기소개를 보통 불렛 형태로 작성하진 않아서 어색해 보입니다. 줄글 형식이 더 적절할 것 같습니다.
	- 자기소개에 키워드가 좀 많은 것 같다. 핵심 키워드로 간결하게 설명해도 좋을 것 같다.
	- 첫번째 문장에서 넥슨에서~ 이 단어 뒤로는 생략 해도 될 것 같다. 자기소개에서는 개발자로써 철학이나 중요시하는 가치, 강점 들을 나타내는 곳이라 생각.
	- 코드의 안정성은 어떤 성과 혹은 효과인데.. 예컨데 제품의 중요한 시나리오 테스트 커버리지를 높여 코드의 안정성을 높인다 혹은 안정적인 코드를 만든다 와 같은게 인과관계가 맞을 것 같다 (명확한 역할은 2번째 문장에서 애매한 항목으로 보인다)
	- 3번째 문장은 차라리 생략하는게 나을 것 같다.
	- 4번째 문장은 개발자로써 "반복 작업" 혹은 "중복"과 같은 것들을 자동화하는 좋은 강점으로 보인다. 
	- 그리고 이 스타일 사실 눈에 잘 안들어온다. 가독성이 좋지 않다. 차라리 볼드나 언더라인으로 더 표현하는게 좋을 것 같다.
	- 키워드를 가독성 높은 코드 작성과 협업 중심 문화를 지향, 코드의 안정성, 불필요한 반복 혹은 중복은 자동화 그래서 이러한 것들을 통해 어떤 성과를 낼 수 있는지 표현하면 좋을 것 같다. (성과는 어떠한 강점에 대한 성과를 작성 작성할 수 없다면 그 자체로 내 가치관을 설명 예컨대 첫번째 문장)

### 3-3. 이력서 총평
- 자기소개 소개처럼 반복 작업 "자동화"와 같은 협업간 혹은 팀내 개선 업무 사항이 많다.
- 주어진 기능 개발 뿐만 아니라 제품 개발 혹은 협업 간에 보이지 않는 문제를 정의하고 그걸 기술로 해결한 사례가 많아서 차별성이 있음
- 아마 서류레벨에서 탈락한 이유는 이러한 이력들이 조금 이해가 안가거나 장황하거나 혹은 도입 이유, 도입 후 성과 같은 것들이 작성되어있지 않아서 일듯. 그래서 단순히 기술 도입 혹은 이해하지 못하고 떨어졌을 확률이 있다.
- 이에대한 것만 개선되면 서류합격은 높아질듯
- 그리고 질문주신 서비스를 위한 인프라 개발 등에 대한건.. 회사 도메인 특성상 개선하기 어려울듯.
- 차라리 코드 레벨에서 중복될 수 있는 것들을 찾아서.. open api generator나 지금 하신 시간 단축 혹은 번역 시트 자동화 등 이런 업무로 기술 이력에 차별성을 두어 다음 회사에서 next를 통한 서버 혹은 인프라까지 업무 범위를 넓혀서 하는게 나으실듯.

- trun
	- _trun에서 담당하신 페이지나 도메인을 작성해주세요. 평가자에게 담당 도메인 지식 및 아래 기술 이력에 대한 배경지식을 제공할 수 있습니다. 제 경력을 예로 들면 ex.) 결제, 뷰어, 인증, 메인홈 구현_
	- **필터링 UI의 상태 관리 방식을 URL 쿼리스트링 기반으로 리팩터링**
		- 필터 상태 관리를 컴포넌트 로컬 상태(useState)에서 **URL 쿼리스트링 기반으로 전환**하여 상태의 출처를 단일화
		- URL 검색 파라미터를 기준으로 필터 상태를 구성하고, 필터 변경 시 URL이 상태의 **source of truth**가 되도록 구조화
		- API 요청 파라미터, 렌더링 상태, URL 간의 불일치를 제거하여 복수 컴포넌트 간 **일관된 상태 동기화**가 가능한 구조로 개선
	- _1,2줄로 줄여도 좋지 않을까요?_
		- 내용 자체는 쉽게 이해됐습니다. 실서비스에서도 해당 방법을 많이 사용하고 있습니다. 
		- 수정 내용은.. 일단 작업(What), 방법(how), 성과(why) 를 잘 써주셨습니다.
		- 위 작업에서 핵심 키워드는 single source of truth 로 구조화하여 단일 상태 흐름 구축 으로 보입니다.
		- 위 키워드를 사용하여 조금 더 컴팩트하게 수정해주시면 좋을 것 같습니다.
	- **인증/UX 흐름 개선 및 전반 구조 정리 기여**
		- API 응답이 401 상태일 때 사용자에게 구별된 상호작용을 제공하기 위해 로그인 모달, 로그아웃 처리 등 인증 UX 흐름 설계
			- _1,2,3번 항목이 전부 별개의 내용이겠죠..?_
			- _일관된 인증 플로우 설계_ 일관된 이라는 키워드로 강조하면 좋을 것 같습니다.
		- 모든 팀원의 통일된 예외처리를 위해 공통 에러 구조로 리팩토링 진행
			- _이 내용은 단순 공통 에러 구조 처리라면 추후에 다른 기술 이력 내용으로 대체 가능해보입니다. (위 항목도 동일)_
		- 공통 프록시(미들웨어) 구조를 도입하여 클라이언트 코드에서 인증 관련 의존성을 최소화, UX 개선
			- _미들웨어는 next.js 인가요?_
				- 이 부분을 좀 더 디벨롭 하면 좋을 것 같습니다.
				- next.js 서버를 리버스 프록시로 활용해 인증/토큰 로직을 미들웨어에 추상화
				- 위와 같은 키워드를 이용하여 디벨롭 하시면 좋을 것 같습니다!
				- 이 경험이 좋은 평가를 받을 수 있는 이유는 FE로 지원했지만 문제해결 바운더리를 서버까지 넓혀서 생각할 줄 아는 역량을 보여줄 수 있는 이유입니다.
	- **스크롤 유무에 따른 레이아웃 시프트 문제 해결**
		- 브라우저 및 OS 환경에 따라 스크롤바 노출 방식 차이로 발생하는 레이아웃 시프트 문제를 분석
		- overflow 조정, 여백 보정 등 기존 방식은 모두 스크롤이 레이아웃을 밀어내는 구조라는 한계 확인
		- 스크롤 영역이 레이아웃에 영향을 주지 않도록 **OverlayScrollbars**를 도입해 오버레이 방식 스크롤 구조로 개선
		- 그 결과 스크롤 유무와 관계없이 일관된 UI 레이아웃을 유지하도록 해결
		- _이 내용이 정확히 이해가 가지 않네요; 뷰 관련 트러블슈팅기 내용으로 보이는데 멘토링 때 문의드리겠습니다._
			- 참고로 뷰 관련 트러블슈팅기는 보여지는 영역이라 텍스트로 설명하기 어려운 경우가 많습니다. 이런 내용은 임팩트가 있는 성과나 난이도가 높은 작업이 아니라면 생략을 하거나.. 블로그에 글을 작성하여 이미지와 함께 이해를 도울 수 있는 방법이 있습니다.
	- **인증 보안 강화를 위한 구조 개선 시도**
		- 기존 백엔드 토큰을 localStorage에 저장하던 방식을 **XSS 위험 관점**에서 검토하고, 쿠키 저장 전략 제안
		- 토큰을 클라이언트에서 직접 관리하지 않기 위해 **httpOnly** 설정이 가능한 NextAuth 기반 구조로 리팩토링
		- 모든 API 호출이 공통 프록시 BFF를 거치도록 설계하여 토큰이 클라이언트에 노출되지 않도록 개선
		- 이후 NextAuth의 설계 목적(소셜 로그인, 백엔드 없는 인증)과 팀의 요구사항이 맞지 않음을 인지하고, **Auth BFF로 재리팩토링**하여 라이브러리 의존성 최소화
	- _일단 쿠키로 전환한 점은 너무 좋습니다._
	- _제목은 '시도' 라는 키워드를 제거해주세요. 시도만 하고 끝난건가? 라는 가능성을 배제하면 좋을 것 같습니다. 해당 경험은 좋은 구조 전환 작업입니다._
	- _이 내용을 보니 BFF로 잘 구현하셨네요 ㅎㅎ NextAuth(시행착오) 내용은 불필요합니다. 이력서에는 작업(What), 적용 방식(How), 성과(Why)를 키워드 위주로 컴팩트하게 작성해주세요._
	- _Auth BFF로 재리팩토링 -> 구체적으로 어떻게 재리팩토링 한건지 작성부탁드립니다. (그리고 재리팩토링 보다는 위에서 강조한 '추상화' 에 초점을 맞추시면 좋을 것 같아요) 구체적인 내용이 없어서 평가하기 어렵습니다. 개인적으로 위 경험의 좋은 점은 localStorage -> Cookie 전환 주도성, 보안까지 고려한 작업, 더 나아가서는 (이걸 하셨는지는 모르겠음) BFF를 API 호출 단일 진입점으로 통합하고, 인증/토큰 관리를 미들웨어 레이어로 추상화' 를 강조하는 방향으로 수정되면 좋을 것 같습니다._
	- _위 내용은 클라쪽 인증 의존성을 제거하여 보안/책임 분리 측면의 장점을 가져가는 좋은 설계로 보입니다._

- 자기소개
	- 자기소개가 조금 긴 것 같습니다.
	- 자기소개는 평가에 크게 영향을 미치지 않습니다. 그래서 큰 시간을 들이지 않고 1문단으로(2~4줄) 간결하게 수정해주셔도 좋을 것 같아요.
	- 자기소개는 2가지 타입으로 쓸 수 있을 것 같아요
	- a. 자신의 기술 이력 내용들을 핵심 키워드로 나타내며 요약하여 소개하는 타입
	- ex.
		- 구조와 흐름을 중심으로 문제를 정의, 상태·인증·UX 관점에서 해법을 설계하는 개발자.
		- 인증 흐름 재설계, BFF 추상화 등 실제 서비스에서 발생한 문제를 구조적으로 개선해 왔습니다.
	- b. 개발자로써 내가 중요하게 생각하는 가치관
	- ex. 
		- 불필요한 작업을 발견하고 이를 개선하는 것을 좋아합니다. 비효율적인 프로세스나 중복 작업을 찾아내어 최적화하거나, 자동화를 통해 생산성을 높이는데 관심이 많다. 이를 통해 팀의 효율성을 향상시키는 것을 목표로 합니다. 
		- 프론트엔드에 국한되지 않고 백엔드, 인프라까지 폭넓게 관심을 가진다. 다양한 직군과 협업하여 복잡한 문제를 해결하고, 그 경험을 사내에 공유하여 조직 전체에 가치를 더한다

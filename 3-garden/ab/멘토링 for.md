### 이력서 리뷰에서 내가 해줘야할 일

- 이력서 순서 중요도

#### 자기소개 리뷰

- 자기소개는 자신이 생각하는 핵심가치와 철학을 작성한다. 
	- "자원을 최대한 활용하여 임팩트를 극대화하는 방법을 고민하지만, 때로는 비효율적인 선택도 필요하다고 믿는다."
- 혹은 전문성과 강점을 드러낸다.
	- 불필요한 작업을 발견하고 이를 개선하는 것을 좋아합니다. 비효율적인 프로세스나 중복 작업을 찾아내어 최적화하거나, 자동화를 통해 생산성을 높이는데 관심이 많다. 이를 통해 팀의 효율성을 향상시키는 것을 목표로 합니다. 
	- 프론트엔드에 국한되지 않고 백엔드, 인프라까지 폭넓게 관심을 가진다. 다양한 직군과 협업하여 복잡한 문제를 해결하고, 그 경험을 사내에 공유하여 조직 전체에 가치를 더한다
	- 안티패턴: React, Nextjs 학습하여 SSR 적용해본 경험이 있다.

##### 요약하자면
자기소개에서는 개발자로써 자신이 중요하게 생각하는 가치와 철학을 나타내거나 전문성이나 강점을 드러내는 곳이라고 생각해주시면 좋을 것 같아요
주니어 혹은 취준생 포지션이더라도 개발자로써 자신의 관심사 중요하게 생각하는 가치와 철학을 고민해보시고 수정해주세요! 이런 부분에서 차별성을 만들어낼 수 있고 이런것들을 프로젝트나 기술이력에서 잘 드러낼 수 있다면 충분히 차별성 있는 이력서가 될 수 있을 것 같아요.

참고로 자기소개에서 작성한 문장들은 블로그, 깃헙, 경력 혹은 프로젝트 사항에서 뒷받침될만한 사항이 있는지 확인합니다. 혹은 면접 단계에서도 확인하구요.
자기소개에 가장 많이 소개되는 문구 중 하나인
최신 기술과 트렌드에 관심이 많고 이를 팀에 공유.. 와 같은 내용이 많아서
이러한 내용이 있을 때 면접에서 최근에 어떤 기술에 관심있었는지 해당 기술에 대해 어떻게 생각하는지 최신 기술을 팔로업하면서 실제로 제품에 적용해본 경험이 있는지 그로인한 성과가 있었는지.. 등 이런식으로 검증 절차를 거치기도 합니다.


#### 이력서 경력사항 리뷰
- 경력사항이 잘 이해가 안되거나 모호한것
	- 이것은 고민하지 말고 바로 멘토링 시간에 질문
- 불필요한 것은 제거 혹은 디벨롭한다.
	- 불필요한 것이란?
		- 단순 적용.
			- ex. react-query를 이용한 API 호출
			- 제거 
			- 만약 개선하고 싶다면 
				- 리액트쿼리를 이용한 캐싱전략을 통해 네트워크 요청 40% 감소
			- 이력서에 UX 개선 항목이 있다면
				- 사용자 피드백 속도 향상을 위한 optimistic update를 활용
			- 팀내 협업 관점에서 중복 제거 및 코드 컨벤션 작업
				- React Query를 사용할 때 queryKey를 고유하게 설정해야 하는 부담을 덜기 위해 API 요청 정보를 기반으로 queryKey를 자동 생성하는 Hook 패키지를 개발
#### 차별성 만들기
- __나는 이런 문제를 해결할 수 있는 사람이다__ 로 요약해드리기
	-  __강점에 집중__ 한 이력서:  의 메시지
- __가장 중요한 방향성__
	- 단순 기술 스택 도입 << 문제 정의 -> 설계 -> 성과(검증)
		- 이 기술스택을 도입한 이유 혹은 성과 작성
		- 의사결정 과정이 보여야한다. (단순 도입 ㄴ)
	- 기능 나열형(요구사항 대로 구현) << 불편함(문제 정의) -> 개선(방법) -> 결과(성과)
		- 이걸 쓰지말라는건 아님. 이걸 해야 문제가 발생함. 이것만 나열하는게 문제.
	- low level(코드 설명), high level(포장만 함) << What, How, Why / 문제, 판단, 결과 압축
	- 개인의 성실함 << 제품, 코드 품질, 자동화 / 병목, 프로세스 비효율을 기술로 해결하기
		- 재발 방지
		- 시스템적 해결
		- 자동화,설계(ex.추상화)를 통해 휴먼 에러 제거
		- 팀 생산성
		- 즉, 사람이 잘해야 유지되는 구조 X, 사람이 실수해도 망가지지 않거나 실수 방지하는 구조
	- 예를 들어보자
		- e2e 테스트 코드 작성하여 버그 선제적 대응
			- 임팩트 X
		- PR 단계에서 주요 유저 시나리오를 e2e 테스트 코드를 통해 품질 검증 자동화
		- backstopJS 기반의 시각적 회귀 테스트를 공통 컴포넌트에 도입하여 리그레션 테스트 비용 감소
	- 즉, 개인의 노력에 의존 X, 시스템에 문제 해결 위임하는 방식
##### 요약하자면
기술 이력 작성에 대해 쪼금 더 고민이 필요해 보여요. 
기술 이력에서 단순 기능 구현은 차별성이 드러나기 힘든 항목 중 하나 입니다. 
차별성이 드러날 수 있는 것들은 성능 개선 혹은 프로젝트 협업 중에 발생한 문제나 중복 프로세스 비효율을 발견하고 이를 개발자로써 해결한 사례가 있는지..

자신만의 시행착오 혹은 의사결정이 보이는 기술이력에 대해 고민이 필요해보입니다.
프로젝트의 기간이 1달밖에 안되다보니 아마 기능 구현하느라 일정이 부족하셨을텐데요. 이제는 기능 구현을 넘어서서 차별성 있는 이력이 필요합니다.
이런 안목을 기르기 위해 큰회사의 기술 블로그(토스, 네이버, 배민, 카카오 등)의 포스팅을 자주 들여다보면서 문제를 발견할 수 있는 안목을 기르고 실제로 프로젝트에 적용해보는 경험이 필요해보입니다. 
자세한 예시는 리뷰 때 설명드릴께요!

### 이력서 리뷰 후에 앞으로의 방향성
대부분의 이력서들은 차별성이 없다. 일단 내가 겪은 유형은 크게 두가지같다.
1. 프론트엔드인데 단순 적용만 한 사례
2. 백엔드인데 프론트엔드 이력이 없는 사례

- [open api generator](https://velog.io/@kdeun1/OpenAPI-Generator%EB%A5%BC-%EC%82%AC%EC%9A%A9%ED%95%98%EC%97%AC-API%EC%99%80-%EB%8F%99%EC%9D%BC%ED%95%9C-Model%EA%B3%BC-%EC%A0%95%ED%98%95%ED%99%94%EB%90%9C-API%EC%BD%94%EB%93%9C-%EC%9E%90%EB%8F%99%EC%83%9D%EC%84%B1%ED%95%98%EA%B8%B0)

두가지 유형 다 FE개발자 경력이 있다면 했으면 좋았을 것들에 대해 소개해주자. 이거는 왜 이걸 해야하는지, 이걸 했을 때 어떤 효과가 있는지에 대해 설명하면 지원자 입장에서 이해하기 쉬울 것 같다.
- 디자인 시스템
	- 디자이너와 개발자의 협업
	- 다양한 컴포넌트의 재사용성을 높이면서 변경을 쉽게하기 위해 추상화 과정을 거친다.
		- 이 추상화과정에 대해 How를 추가한다면 +
			- ex. 컴포넌트의 추상화 수준 정의를 4가지로 정의 
				- 도메인 컴포넌트를 조합한 컴포넌트, 도메인 컴포넌트, 제품시스템, 디자인시스템
	- 동작의 일관성을 보장하기 위한 헤드리스는 덤
	- 그리고 컴포넌트의 리팩토링 혹은 변경시 기존 동작 안정성 보장을 위한 시나리오 기반 __테스트__ 구축 경험
- 테스트 코드 작성 (e2e, 통합, 유닛)
	- WHY
		- 리팩토링 혹은 변경시 예상치못한 사이드 펙트를 사전에 방지 + 기존 동작 확인
		- 테스트 코드를 작성하다보면 코드의 모듈화나 재사용성을 개선한다
			- 1. 테스트 코드를 작성하다보면 테스트가 복잡해지는 걸 느낄 수 있다.(테스트 )
				- 이건 컴포넌트가 SRP를 지키지 못한다는 것
				- 그래서 컴포넌트를 SRP에 맞게 수정
			- 2. 의존성을 낮출 수 있다
				- 특정 컴포넌트가 어떤 함수나 의존하게 된다면 
		- 테스트 코드는 스펙 문서가 될 수 있다.
- 개발 과정 중 Blocking Point 찾아내서 제거하기
	- API 배포가 항상 늦는 문제 즉 협업간 발생하는 이슈를 개발로 어느정도 해결한 방법
		- 예컨대 API와 클라이언트간 병렬 작업 환경 구축: API 배포까지 기다리면 일정 낭비
			- MSW, Proxy 구성
				- MSW
					- local, dev -> MSW / qa, production -> API
				- Proxy
					- Client -> Next.js Proxy -> 실제 API
						- Client는 동일한 엔드포인트를 호출. Proxy에서 Mock, API 분리
						- if문 분기가 아닌 Proxy레벨에서 처리(추상화 효과)
- 설계, 개발, 배포, 운영까지 경험 보유
	- CI, CD를 위한 gitops & k8s
- nextjs 부하 테스트
	- 스트레스 테스트 툴을 사용한 TPS 개선기: SSR, SSG 적절한 활용, 캐싱, 네트워크최적화 등
- 소프트스킬 
	- 가장 좋은건 팀 미션 달성하기
		- 동료들과 논의하며 비즈니스 임팩트 만들어내기
- 주의할점
	- 성능 개선 외에 개선할 수 있는 것들이 많다. 개발 과정 중 Blocking Point 를 찾아내자.

대화 정리: 프롬프트 파일 참조 및 규칙 준수 전략

  📌 최초 질문

  "프롬프트 파일에서 다른 파일의 규칙을 준수하도록 Claude에게 명령하는 가장 효과적인 방법은?"

  ---
  🔍 논의된 주요 질문들

  1️⃣ 예제 코드를 프롬프트에 포함시키는 이유

  질문: "너는 기본 패턴을 알고 있는데 왜 예제 코드를 추가하나?"

  답변:
  - ✅ 포함 필요: 프로젝트 고유 패턴 (예: mx-11pxr, font-large4-bold)
  - ❌ 불필요: 일반적 패턴 (예: useState, import 문법)

  판단 기준:
  "이 프로젝트만의 특별한 패턴"이거나 "판단이 필요한 선택지"일 때만 예시 포함

  ---
  2️⃣ 파일 참조 vs 내용 통합

  문서 참조 방식:
  **@figma-to-code-conversion.md 전체 준수**

  3가지 문제점:
  1. ❌ "언제 읽을지" 제가 판단 → 늦게 읽음
  2. ❌ "어떻게 해석할지" 제가 판단 → 오해 가능
  3. ❌ 실행 흐름 복잡 → 여러 단계

  내용을 전부 프롬프트에 통합하면?
  - ⚠️ 문제 해결 안 됨: 50개 규칙을 평면적으로 나열하면 우선순위 파악 어려움
  - 🔑 핵심: 위치(파일 vs 인라인)가 아니라 구조화가 중요

  ---
  3️⃣ 타이밍 명시의 효과

  단순 타이밍 추가:
  구현 시작 전 필수: @some_file.md
  막혔을 때 참고: @some_file.md

  개선 효과:
  - ✅ "언제 읽을지" 문제 해결
  - ❌ "어떻게 해석할지" 여전히 문제
  - ❌ "50개 규칙 우선순위" 여전히 문제

  결론: 타이밍만으로는 30% 개선

  ---
  4️⃣ 핵심 문제: TOP 3만 강조하면 나머지 규칙 위반

  당신의 우려 (정확함!):
  TOP 3 강조 → 규칙 1,2,3 준수 ✅
           → 규칙 4,5,6 위반 ❌
           → 사람이 수정 필요
           → 자동화 실패!

  해결책: 자동 검증 루프
  빠른 구현 (TOP 3) → 전체 규칙 자동 체크 (50개) → 자동 수정 → 재검증 → 완료

  ---
  5️⃣ Sub-agent/Skills는 해결책인가?

  답변: ❌ 아니요

  이유:
  - Sub-agent도 똑같은 문제 겪음: "50개를 어떻게 체크하지?"
  - 문제를 다른 agent에게 넘긴 것뿐

  Sub-agent 유용한 경우:
  - 병렬 작업 (여러 컴포넌트 동시 구현)
  - 독립적 리서치

  이 케이스:
  - 순차적 검증 (50개 규칙)
  - 명확한 체크리스트가 답

  ---
  🎯 최종 권장 전략: 3-Layer 구조

  ┌─────────────────────────────────────┐
  │ Layer 1: TOP 3-5 핵심 규칙 (인라인) │  ← 즉시 적용
  │ - 가장 중요하고 자주 틀리는 것      │
  │ - 프로젝트 고유 패턴                │
  │ - 코드 예시 포함                    │
  └─────────────────────────────────────┘
                ↓
  ┌─────────────────────────────────────┐
  │ Layer 2: 체크리스트 (명확한 단계)   │  ← 전체 규칙 보장
  │ - 구현 전 필수 확인 (4-5개)        │
  │ - 자동 검증 방법 (grep, 비교 등)   │
  │ - 반복 루프 명시                    │
  └─────────────────────────────────────┘
                ↓
  ┌─────────────────────────────────────┐
  │ Layer 3: 전체 참조 문서             │  ← 상세 스펙
  │ - 구체적 섹션 링크                  │
  │ - 막혔을 때 참고                    │
  └─────────────────────────────────────┘

  ---
  📝 구체적 작성 패턴

  ### N단계: 컴포넌트 구현

  #### 즉시 적용 핵심 규칙 (TOP 3):
  ```tsx
  // 1. pxr 단위 (가장 자주 틀림!)
  className="w-100pxr h-50pxr"

  // 2. 색상 토큰 (하드코딩 금지!)
  className="bg-el-10 text-el-70"

  // 3. presentational 패턴 (카카오페이지 표준)
  <Card className="mx-11pxr w-auto">
    <Text className="font-large4-bold">
      {t('key')}
    </Text>
  </Card>

  구현 시작 전 필수 체크리스트:

  @figma-to-code-conversion.md를 읽고 아래 확인:
  - pxr 변환 규칙 이해
  - 색상 토큰 매핑 확인
  - 반응형 breakpoint 규칙 확인
  - 간격 측정 방법 숙지

  구현 후 자동 검증 (전체 규칙 보장):

  1. pxr 규칙 체크:
  grep -n '[0-9]px[^r]' ComponentName.tsx
  2. 발견 시 → pxr로 자동 수정
  3. 색상 규칙 체크:
  grep -n '#[0-9A-Fa-f]' ComponentName.tsx
  4. 발견 시 → 토큰으로 변환

  (... 50개 규칙 자동 체크 방법)

  5. Figma 비교 체크:
  [기존 6단계 검증 프로세스]
  6. 모든 규칙 ✅ 될 때까지 1번부터 반복

  막혔을 때 참고:

  - 반응형: @figma-to-code-conversion.md의 "breakpoint 규칙" 섹션
  - 다크모드: @figma-to-code-conversion.md의 "다크모드 토큰" 섹션

  ## ✅ 핵심 원칙 정리

  ### 1. 예제 코드 포함 기준
  ✅ 프로젝트 고유 패턴
  ✅ 토큰/시간 효율성
  ✅ 명확한 의도 전달
  ❌ 일반적 프로그래밍 패턴

  ### 2. 파일 참조 개선 방법
  ❌ "전체 준수"만 쓰기
  ✅ 구체적 섹션 지정
  ✅ 타이밍 명시 (구현 전, 막혔을 때)
  ✅ 핵심 규칙 인라인 추가

  ### 3. 전체 규칙 준수 보장
  ❌ TOP 3만 강조
  ❌ Sub-agent에게 위임
  ✅ 자동 검증 체크리스트
  ✅ 반복 루프 (98% 일치까지)

  ### 4. 구조화가 핵심
  - 위치(파일/인라인) < **구조화 방식**
  - 평면 나열 < **계층적 구조**
  - "전체 보라" < **"TOP 3 → 체크리스트 → 전체"**

  | 방식 | 타이밍 | 우선순위 | 전체 준수 | 자동화 수준 |
  |------|--------|----------|-----------|------------|
  | 파일 참조 (현재) | ❌ 늦음 | ❌ 불명확 | ⚠️ 놓칠 수 있음 | 60% |
  | 타이밍만 추가 | ✅ 즉시 | ❌ 불명확 | ⚠️ 놓칠 수 있음 | 70% |
  | 3-Layer 구조 | ✅ 즉시 | ✅ 명확 | ✅ 체크리스트 보장 | 95% |

  ## 📚 참고 자료

  **Anthropic 공식 문서**:
  - "Be Clear and Direct" - 동료가 이해 못하면 Claude도 못함
  - "Use Examples" - 예시는 매우 강력함
  - 10만 토큰 이하는 명확성 > 위치

  **결론**: 파일 참조를 버리지 말고, **구조화 + 명확한 타이밍 + 체크리스트**로 개선!

### 왜 그냥 참조만 하면 문제가 발생하는가?

```
#### 4-1. 기본 변환 규칙 적용

**참고**: 기존 presentational 컴포넌트(`apps/kakaopage/src/domain/gacha/presentational/` 등) 구현 패턴 참고
```

일단 의도자체는 난 저 파일을 다른 프롬프트에서도 사용하려고 작성. 그리고 매번 저 규칙을 작성한다는게 누락 혹은 중복의 가능성이 있기 때문.

하지만.. 클로드는 위 패턴을 추천하지 않는데 이유는 다음과 같다.

```
 문제 1: "언제 읽을지" 판단

  1. 프롬프트 읽음: "기존 컴포넌트 패턴 참고하라고?"
  2. 판단: "지금 읽을까? 나중에 읽을까?"
     - 옵션 A: "일단 Figma부터 보고, 막히면 그때 봐야지"
     - 옵션 B: "미리 읽어두고 패턴 파악해야겠다"
  3. 선택: 보통 **옵션 A** 선택 (늦게 읽음)
  4. 결과: Figma 분석 → 코드 작성 시작 → "어? 이거 어떻게 쓰지?" → 🔄 뒤늦게 파일 읽으러 감

  ❌ 문제 2: "어떻게 해석할지" 판단

  1. 기존 컴포넌트 읽음:
     - RouletteButton.tsx: `className={twMerge('h-48pxr', className)}`
     - PasswordSetup.tsx: `<Card className="mx-11pxr w-auto">`

  2. 해석 고민:
     - "패턴이 뭐지? twMerge 쓰는 게 패턴?"
     - "mx-11pxr이 패턴? 아니면 Card 쓰는 게 패턴?"
     - "어떤 파일의 어떤 부분을 따라야 하지?"

  3. 결과: **제 나름대로 해석**해서 적용 → 의도와 다를 수 있음

  ❌ 문제 3: "실행 흐름 복잡"

  프롬프트 읽기
    → Figma 분석
    → "기존 패턴 뭐지?"
    → Glob으로 파일 검색
    → Read로 여러 파일 읽기
    → 패턴 추론
    → 구현 시작
    → "이게 맞나?" 의심
    → 다시 다른 파일 읽기
    → ...
  → 7-8 단계, 불확실한 흐름

  ---
  개선 방식: 핵심 패턴 명시

  #### 4-1. 기본 구현 패턴

  **필수 패턴**:
  ```tsx
  // 1. className props 받고 twMerge로 병합
  export function ComponentName({className, ...props}: Props) {
    return (
      <div className={twMerge(
        'flex flex-col',      // 기본 스타일
        'mx-11pxr',           // 카카오페이지 표준 마진
        className             // 외부 override
      )}>
        {/* ... */}
      </div>
    );
  }

  // 2. Card 래퍼 사용
  <Card className="mx-11pxr w-auto">
    <Text className="font-large4-bold text-el-70">
      {t('translation_key')}
    </Text>
  </Card>

  상세 규칙: @apps/kakaopage/docs/playbook/official/figma-to-code-conversion.md
  참고 예시: apps/kakaopage/src/domain/gacha/presentational/

  **제 실제 행동**:

  #### ✅ 해결 1: 즉시 읽음
  4. 프롬프트 읽음: "아, twMerge 쓰고, mx-11pxr 쓰고, Card 이렇게 쓰면 되겠구나"
  5. 즉시 패턴 파악 완료
  6. 바로 구현 시작

  #### ✅ 해결 2: 명확한 해석
  7. "twMerge로 className 병합" ← 명확
  8. "mx-11pxr이 표준 마진" ← 명확
  9. "Card는 이렇게 감싸기" ← 명확
  10. 해석 불필요, 그대로 적용

  #### ✅ 해결 3: 단순한 흐름
  프롬프트 읽기
    → 패턴 즉시 파악
    → Figma 분석
    → 패턴 적용해서 구현
    → 완료
  → **4 단계**, 확실한 흐름
```

### 위와 관련된 문서 링크
- [Prompt Engineering Guide](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/overview#before-prompt-engineering)
	- Be Clear and Direct
		- 동료가 이해 못하면 클로드도 못한다
	- Use Examples
		- __예시는 강력하다__
- [Long Context](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/long-context-tips)
	- 긴 문서는 프롬프트 앞쪽에 배치
		- 클로드가 더 효과적으로 처리함
		- 하지만! 10만 토큰 이하는 명확성이 위치보다 중요

- [GPT best practice](https://platform.openai.com/docs/guides/prompt-engineering)
	- 참조 텍스트 제공하되, using the provided text처럼 명시적 지시 필요
		- 난해한 주제, 출처 요구받을 때 허구의 답변(환각) 자신있게 만들어냄. 참조용 텍스트 제공하면 그 텍스트 기반으로 답변하도록 지시

## 🔑 핵심 원칙 정리

  ### Anthropic이 말하는 좋은 프롬프트:
  1. **Be Clear and Direct** - 동료가 이해할 수 있게
  2. **Use Examples** - 예시는 강력하다
  3. **Give Claude Time to Think** - 단계별 지시

  ### 파일 참조를 명확하게 만드는 법:
  4. **"전체 준수" 금지** → "이 3개 섹션 읽어" 명시
  5. **읽을 타이밍 지정** → "구현 전", "막혔을 때"
  6. **핵심 3-5개는 인라인** → 즉시 이해 가능하게
  7. **참조는 구체적 섹션** → "문서 전체" 아닌 "X 섹션"
